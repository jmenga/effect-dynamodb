Build a production-grade REST API for managing cricket matches, teams, players, and more — all backed by a single DynamoDB table. You'll learn effect-dynamodb from the ground up: models, entities, queries, refs, cascades, and aggregates.

**What you'll build:** A cricket match management system with:
- 5 standalone entities (Team, Player, Coach, Umpire, Venue)
- Entity references with cascade updates (SquadSelection)
- A complex Match aggregate with sub-aggregates (TeamSheet, UmpireSheet)
- Full CRUD REST API with pagination, error handling, and validation

**How this tutorial works:** Rather than building all models, then all entities, then all services — we'll build one entity end-to-end, get the server running, and validate with curl. Then we'll add more entities iteratively, testing as we go.

**Prerequisites:**
- Node.js 20+
- pnpm 9+
- Docker (for DynamoDB Local)
- Basic familiarity with TypeScript and Effect

## Phase 2: Your First Entity End-to-End — Team

Rather than defining all models up front, we'll build one entity from model to running API, validating each step along the way.

### 2.1 Branded ID type

Create branded types for entity IDs. We only need `TeamId` for now — we'll add more as we build more entities:

```ts
// src/models/Ids.ts
import { Schema } from "effect"

export const TeamId = Schema.String.pipe(Schema.brand("TeamId"))
export type TeamId = typeof TeamId.Type
```

> **Concept: Schema.brand** — Creates a nominal type that is structurally a `string` but branded at the type level. `TeamId` and `PlayerId` are both strings at runtime, but TypeScript treats them as incompatible types. This catches mix-ups at compile time.
>
> These branded types flow through the entire stack: `Entity.Input<typeof Teams>` has `id: TeamId`, `Entity.Key<typeof Teams>` has `id: TeamId`, and ref input types like `Entity.Input<typeof SquadSelections>` have `teamId: TeamId` and `playerId: PlayerId` — ensuring type safety from models through services to API handlers. Each branded ID also doubles as a Schema — use `TeamId` directly in `HttpApiEndpoint` params to decode URL strings into branded types at the API boundary.

### 2.2 Enums

```ts region="model-enums" example="cricket.ts"
const PlayerRole = {
  Batter: "batter",
  Bowler: "bowler",
  AllRounder: "all-rounder",
  WicketKeeper: "wicket-keeper",
} as const
const PlayerRoleSchema = Schema.Literals(Object.values(PlayerRole))
```

> **Pattern: Enum objects with Schema.Literals** — Define enums as `const` objects for autocomplete and referenceability (`PlayerRole.Batter`), then derive a `Schema.Literals` for runtime validation. This gives you two things: a namespace (`PlayerRole.Batter`) and a schema (`PlayerRoleSchema`).

### 2.3 Team model

```ts region="model-standalone" example="cricket.ts"
class Team extends Schema.Class<Team>("Team")({
  id: Schema.String.pipe(DynamoModel.identifier),
  name: Schema.String,
  country: Schema.String,
  ranking: Schema.Number,
}) {}

class Player extends Schema.Class<Player>("Player")({
  id: Schema.String.pipe(DynamoModel.identifier),
  firstName: Schema.String,
  lastName: Schema.String,
  role: PlayerRoleSchema,
}) {}

class Coach extends Schema.Class<Coach>("Coach")({
  id: Schema.String.pipe(DynamoModel.identifier),
  name: Schema.String,
}) {}

class Venue extends Schema.Class<Venue>("Venue")({
  id: Schema.String.pipe(DynamoModel.identifier),
  name: Schema.String,
  city: Schema.String,
}) {}
```

Two things to notice:

1. **`Schema.Class`** — Defines a runtime-validated class. `new Team({ id: "aus", name: "Australia", country: "Australia", ranking: 1 })` validates all fields.

2. **`DynamoModel.identifier`** — Marks the `id` field as the primary business identifier. Other entities can reference this entity by this field. The model stays pure otherwise -- no DynamoDB concepts.

3. **Pure domain model** — No DynamoDB imports or annotations beyond the identifier marker. The model is portable and testable — DynamoDB binding happens later at the entity level.

### 2.4 Teams entity

The entity binds your pure model to DynamoDB with indexes, key composition, and optional features:

```ts region="entity-standalone" example="cricket.ts"
const Teams = Entity.make({
  model: DynamoModel.configure(Team, { id: { field: "teamId" } }),
  entityType: "Team",
  indexes: {
    primary: {
      pk: { field: "pk", composite: ["id"] },
      sk: { field: "sk", composite: [] },
    },
  },
})

const Players = Entity.make({
  model: DynamoModel.configure(Player, { id: { field: "playerId" } }),
  entityType: "Player",
  indexes: {
    primary: {
      pk: { field: "pk", composite: ["id"] },
      sk: { field: "sk", composite: [] },
    },
  },
})

const Coaches = Entity.make({
  model: DynamoModel.configure(Coach, { id: { field: "coachId" } }),
  entityType: "Coach",
  indexes: {
    primary: {
      pk: { field: "pk", composite: ["id"] },
      sk: { field: "sk", composite: [] },
    },
  },
})

const Venues = Entity.make({
  model: DynamoModel.configure(Venue, { id: { field: "venueId" } }),
  entityType: "Venue",
  indexes: {
    primary: {
      pk: { field: "pk", composite: ["id"] },
      sk: { field: "sk", composite: [] },
    },
  },
})
```

Let's break down what each piece does:

- **`DynamoModel.configure(Team, { id: { field: "teamId" } })`** — Renames the `id` field to `teamId` in the DynamoDB item, avoiding collisions when multiple entity types share a table. Since the model already marks `id` with `DynamoModel.identifier`, the entity inherits that for ref resolution.

- **`entityType: "Team"`** — A discriminator value stored as `__edd_e__` on every item. Used to identify which entity type an item belongs to during queries.

- **`indexes.primary`** — The table's primary key. `pk` is composed from the `id` field. `sk` has an empty composite (the library fills it with the entity type).

> **Key concept:** `Entity.make` returns a **pure entity definition** — no table reference, no executable operations. The entity declares its model, indexes, and configuration. Table association happens when you register it on a `Table.make({ entities: { Teams } })`, which we'll create next.

### 2.5 Schema and Table

Now that we have an entity, we need a **DynamoSchema** (application namespace) and a **Table** (physical table binding) to assemble it:

```ts region="schema" example="cricket.ts"
const CricketSchema = DynamoSchema.make({ name: "cricket", version: 1 })
```

```ts region="table" example="cricket.ts"
const MainTable = Table.make({
  schema: CricketSchema,
  entities: { Teams, Players, Coaches, Venues, SquadSelections },
})
```

> **Concept: Single-table design** — All entities share one DynamoDB table. The schema's `name` and `version` become part of every key prefix (`$cricket#v1#...`), preventing collisions between entity types. `Table.make` registers entities up front — you'll add more to the `entities` record as you create them throughout this tutorial.

> **Note:** Entity files don't import the table. Entities are pure definitions. The dependency flows the other way: `Schema.ts` imports entities and registers them on the table. In the service layer, `DynamoClient.make(MainTable)` binds all registered entities and returns a typed client where every operation has `R = never`.

Run `pnpm check` — the model, entity, and table should compile.

### 2.6 Pagination helpers

Before building the service, create shared pagination schemas that will be reused by every entity:

```ts
// src/models/Pagination.ts
import { Query } from "@effect-dynamodb/core"
import { Schema } from "effect"
import { identity } from "effect/Function"

// ---------------------------------------------------------------------------
// Query string schema — spread `.fields` into HttpApiEndpoint query definitions
// ---------------------------------------------------------------------------

export const PaginationOptions = Schema.Struct({
  cursor: Schema.optionalKey(Schema.String),
  count: Schema.optionalKey(Schema.NumberFromString),
  reverse: Schema.optionalKey(Schema.String),
})

export type PaginationOptions = typeof PaginationOptions.Type

// ---------------------------------------------------------------------------
// Response envelope — schema factory for paginated list responses
// ---------------------------------------------------------------------------

/** Schema factory: creates a paginated response schema for any item type. */
export const PaginatedResponse = <A>(item: Schema.Schema<A>) =>
  Schema.Struct({
    data: Schema.Array(item),
    count: Schema.Number,
    cursor: Schema.NullOr(Schema.String),
  })

// ---------------------------------------------------------------------------
// Query helper — apply decoded query params directly to a Query
// ---------------------------------------------------------------------------

/** Apply PaginationOptions to a Query: sets limit, order, and cursor. */
export const applyPagination = <A>(
  query: Query.Query<A>,
  options?: PaginationOptions,
): Query.Query<A> =>
  query.pipe(
    Query.limit(options?.count ?? 10),
    options?.reverse === "true" ? Query.reverse : identity,
    options?.cursor ? Query.startFrom(options.cursor) : identity,
  )
```

Three things to notice:

1. **`PaginationOptions`** — A `Schema.Struct` that doubles as both schema and type. Spread `PaginationOptions.fields` into endpoint `query` definitions. The HTTP framework decodes query strings (e.g., `?count=20&reverse=true`) into typed values automatically.

2. **`PaginatedResponse(item)`** — A schema factory for runtime use (e.g., in service implementations). **Do not use in `HttpApiEndpoint` definitions** — the generic function indirection exceeds TypeScript's conditional type inference depth inside `HttpApiBuilder.group`, causing the layer's `R` parameter to resolve to `unknown` instead of `never`. Use inline `Schema.Struct({ data: Schema.Array(Entity), count: Schema.Number, cursor: Schema.NullOr(Schema.String) })` in endpoint `success` fields instead.

3. **`applyPagination`** — Works directly with decoded `PaginationOptions`. No manual parsing — the Schema layer already decoded `count` from string to number via `Schema.NumberFromString`.

> **Concept: Query combinators** — `Query.limit`, `Query.reverse`, and `Query.startFrom` are composable functions that modify a query description. Nothing executes until you call `Query.execute` or `Query.collect`.

### 2.7 List filter schema

Each entity's list endpoint has its own filter fields. Define these as `Schema.Struct` values — the same schema drives the API endpoint query definition and the service's filter type:

```ts
// src/models/Filters.ts
import { Schema } from "effect"
import { GenderQuery } from "./Enums.js"

export const TeamListFilter = Schema.Struct({
```

> **Pattern: Schema as value + type** — Exporting `const Foo = Schema.Struct(...)` and `type Foo = typeof Foo.Type` under the same name gives you a schema (for validation and `.fields` spreading) and a TypeScript type (for function signatures) from one definition.

### 2.8 TeamService

```ts
// src/services/TeamService.ts
import { DynamoClient, Entity } from "@effect-dynamodb/core"
import { type DateTime, Effect, ServiceMap } from "effect"
import { ulid } from "ulid"
import { MainTable } from "../entities/Schema.js"
import { Teams } from "../entities/Teams.js"
import type { TeamListFilter } from "../models/Filters.js"
import type { TeamId } from "../models/Ids.js"
import { applyPagination, type PaginationOptions } from "../models/Pagination.js"

export type { DateTime }

type CreateTeamInput = Entity.Create<typeof Teams>
type UpdateTeamInput = Entity.Update<typeof Teams>

export class TeamService extends ServiceMap.Service<TeamService>()("@gamemanager/TeamService", {
  make: Effect.gen(function* () {
    const { Teams: teams } = yield* DynamoClient.make(MainTable)
    return {
      create: Effect.fn(function* (input: CreateTeamInput) {
        const id = ulid() as TeamId
        return yield* teams.put({ ...input, id })
      }),

      get: (id: TeamId) => teams.get({ id }),

      update: (id: TeamId, updates: UpdateTeamInput) => teams.update({ id }, Entity.set(updates)),

      delete: (id: TeamId) => teams.delete({ id }),

      list: (filter: TeamListFilter = {}, pagination?: PaginationOptions) =>
        applyPagination(Teams.query.byAll(filter), pagination).pipe(
          Query.execute,
          Effect.map((page) => ({
            data: page.items,
            count: page.items.length,
            cursor: page.cursor,
          })),
        ),
    }
  }),
}) {}
```

Key patterns:

- **`ServiceMap.Service`** — Declares a service with a tag and a `make` effect that produces the implementation. Consumers access it via `yield* TeamService`.

- **`Entity.Create<typeof Teams>`** — Omits the identifier field (declared via `DynamoModel.configure`) from the input type — the common "create" payload where IDs are auto-generated.

- **`Entity.Update<typeof Teams>`** — Partial type excluding primary key composites. Only the fields you want to change.

- **`ulid() as TeamId`** — `ulid()` returns `string`, so we cast it to the branded type. This is the one place you'll need a cast — at the generation boundary where IDs are created.

- **`DynamoClient.make(MainTable)`** — The typed execution gateway. Resolves `DynamoClient | TableConfig` from context, binds all entities registered on the table, and returns a typed client where every operation has `R = never`. Destructure to get bound entities: `const { Teams: teams } = yield* DynamoClient.make(MainTable)`.

- **Bound entity methods** — `teams.get()`, `teams.update()`, `teams.delete()` return `Effect` with `R = never` — no context requirements.

- **`Query.execute`** — Pipe terminal that executes the query and returns `{ items, cursor }` for pagination. Resolves `DynamoClient | TableConfig` from context.

### 2.9 API error types

```ts
// src/api/ApiErrors.ts
import { Schema } from "effect"

export class ApiNotFound extends Schema.ErrorClass<ApiNotFound>("ApiNotFound")({
  error: Schema.Literal("NotFound"),
  message: Schema.String,
}, { httpApiStatus: 404 }) {}

export class ApiBadRequest extends Schema.ErrorClass<ApiBadRequest>("ApiBadRequest")({
  error: Schema.Literal("BadRequest"),
  message: Schema.String,
}, { httpApiStatus: 400 }) {}

export class ApiConflict extends Schema.ErrorClass<ApiConflict>("ApiConflict")({
  error: Schema.Literal("Conflict"),
  message: Schema.String,
}, { httpApiStatus: 409 }) {}

export class ApiUnprocessableEntity extends Schema.ErrorClass<ApiUnprocessableEntity>("ApiUnprocessableEntity")({
  error: Schema.Literal("UnprocessableEntity"),
  message: Schema.String,
}, { httpApiStatus: 422 }) {}
```

> **Concept: Schema.ErrorClass for API errors** — Each error uses `Schema.ErrorClass` — the idiomatic Effect v4 pattern for HTTP API errors. The second argument sets class-level annotations including `httpApiStatus`. This gives you proper Error instances with constructors (`new ApiNotFound({ error: "NotFound", message: "..." })`), clean JSON responses (no `_tag` in the wire format), and status-code-based discrimination — the HttpApi client dispatches by HTTP status code, not by `_tag`, so each error just needs a unique status.

### 2.10 TeamsApi endpoint definitions

```ts
// src/api/TeamsApi.ts
import { Schema } from "effect"
import { HttpApiEndpoint, HttpApiGroup } from "effect/unstable/httpapi"
import { Teams } from "../entities/Teams.js"
import { TeamListFilter } from "../models/Filters.js"
import { TeamId } from "../models/Ids.js"
import { PaginationOptions } from "../models/Pagination.js"
import { Team } from "../models/Team.js"
import { ApiConflict, ApiNotFound } from "./ApiErrors.js"

export const TeamsApi = HttpApiGroup.make("teams").add(
  HttpApiEndpoint.get("list", "/teams", {
    query: { ...TeamListFilter.fields, ...PaginationOptions.fields },
    success: Schema.Struct({
      data: Schema.Array(Team),
      count: Schema.Number,
      cursor: Schema.NullOr(Schema.String),
    }),
  }),
  HttpApiEndpoint.get("get", "/teams/:id", {
    params: { id: TeamId },
    success: Team,
    error: ApiNotFound,
  }),
  HttpApiEndpoint.post("create", "/teams", {
    payload: Teams.createSchema,
    success: Team,
    error: ApiConflict,
  }),
  HttpApiEndpoint.put("update", "/teams/:id", {
    params: { id: TeamId },
    payload: Teams.updateSchema,
    success: Team,
    error: ApiNotFound,
  }),
  HttpApiEndpoint.delete("remove", "/teams/:id", {
    params: { id: TeamId },
    error: ApiNotFound,
  }),
)
```

> **Concept: HttpApiGroup / HttpApiEndpoint** — Declarative API definition. Each endpoint specifies its method, path, query/params/payload schemas, success type, and error types. The framework validates inputs and serializes outputs automatically.

> **`Teams.createSchema` and `Teams.updateSchema`** — Every entity exposes typed schemas derived from its model. `createSchema` is the input schema minus primary key composites (the common pattern where IDs are auto-generated). `updateSchema` makes all non-PK fields optional. Both preserve branded types — ref ID fields use `PlayerId`, `TeamId`, etc., not plain `string`. Use these directly as `HttpApiEndpoint` payloads to avoid manually re-specifying field schemas.

### 2.11 API composition and handler wiring

```ts
// src/api/Api.ts
import type { Schema } from "effect"
import { HttpApi, OpenApi } from "effect/unstable/httpapi"
import { TeamsApi } from "./TeamsApi.js"

export type { Schema }

export const GameManagerApi = HttpApi.make("gamemanager")

  .annotate(OpenApi.Version, "2.0.0")
```

```ts
// src/api/App.ts
import type { ItemNotFound, UniqueConstraintViolation, Table } from "@effect-dynamodb/core"
import { type DateTime, Effect, Layer } from "effect"
import { HttpServer } from "effect/unstable/http"
import { HttpApiBuilder } from "effect/unstable/httpapi"
import { Team } from "../models/Team.js"
import { TeamService } from "../services/TeamService.js"
import { GameManagerApi } from "./Api.js"
import { ApiConflict, ApiNotFound } from "./ApiErrors.js"

export type { Table, DateTime }

// ---------------------------------------------------------------------------
// Error helpers: catch domain errors, convert to HTTP errors
// ---------------------------------------------------------------------------

const entityNotFound =
  (entity: string, id: string) =>
  <A, E, R>(self: Effect.Effect<A, E | ItemNotFound, R>) =>
    Effect.catchTag(
      self,
      "ItemNotFound",
      () => Effect.fail(new ApiNotFound({ error: "NotFound", message: `${entity} not found: ${id}` })),
      (e) => Effect.die(e),
    )

const uniqueConflict =
  (entity: string) =>
  <A, E, R>(self: Effect.Effect<A, E | UniqueConstraintViolation, R>) =>
    Effect.catchTag(
      self,
      "UniqueConstraintViolation",
      () => Effect.fail(new ApiConflict({ error: "Conflict", message: `${entity} already exists (unique constraint violation)` })),
      (e) => Effect.die(e),
    )

// ---------------------------------------------------------------------------
// Handlers
// ---------------------------------------------------------------------------

const TeamsHandler = HttpApiBuilder.group(
  GameManagerApi,
  "teams",
  Effect.fn(function* (handlers) {
    const svc = yield* TeamService
    return handlers
      .handle("list", ({ query }) =>
        Effect.gen(function* () {
          const filter = query.gender ? { gender: query.gender } : undefined
          const result = yield* svc.list(filter, query)
          return {
            data: result.data.map((r) => new Team(r)),
            count: result.count,
            cursor: result.cursor,
          }
        }).pipe(Effect.orDie),
      )
      .handle("get", ({ params: { id } }) =>
        svc.get(id).pipe(entityNotFound("Team", id)),
      )
      .handle("create", ({ payload }) =>
        svc.create(payload).pipe(uniqueConflict("Team")),
      )
      .handle("update", ({ params: { id }, payload }) =>
        svc.update(id, payload).pipe(entityNotFound("Team", id)),
      )
      .handle("remove", ({ params: { id } }) =>
        svc.delete(id).pipe(entityNotFound("Team", id)),
      )
  }),
)

// ---------------------------------------------------------------------------
// Layer assembly
// ---------------------------------------------------------------------------

const HandlerLayers = Layer.mergeAll(
  TeamsHandler,
)

const ServiceLayers = Layer.mergeAll(
  Layer.effect(TeamService, TeamService.make),
)

export { GameManagerApi } from "./Api.js"

export const ApiLayer = HttpApiBuilder.layer(GameManagerApi).pipe(
  Layer.provide(HandlerLayers),
  Layer.provide(ServiceLayers),
  Layer.provide(HttpServer.layerServices),
)
```

> **Concept: Layer assembly** — Services are composed into layers. `Layer.effect(TeamService, TeamService.make)` creates a layer that runs `TeamService.make` and provides the result. `Layer.mergeAll` combines independent layers. `Layer.provide` feeds dependencies downward.

```ts
// src/server.ts
import { createServer } from "node:http"
import { NodeHttpServer, NodeRuntime } from "@effect/platform-node"
import { DynamoClient } from "@effect-dynamodb/core"
import { Effect, Layer } from "effect"
import { HttpMiddleware, HttpRouter, HttpServerRequest } from "effect/unstable/http"
import { ApiLayer } from "./api/App.js"
import { MainTable } from "./entities/Schema.js"

// Infrastructure layers
const DynamoLayer = DynamoClient.layer({
  region: "local",
  endpoint: "http://localhost:8000",
  credentials: { accessKeyId: "local", secretAccessKey: "local" },
})

const TableLayer = MainTable.layer({ name: "gamemanager" })
const HttpLive = NodeHttpServer.layer(() => createServer(), { port: 3000 })

// Simple request logger
const requestLogger = HttpMiddleware.make((httpApp) =>
  Effect.gen(function* () {
    const request = yield* HttpServerRequest.HttpServerRequest
    yield* Effect.logInfo(`>>> ${request.method} ${request.url}`)
    return yield* httpApp
  }),
)

// Compose and launch
const AppWithInfra = Layer.provide(ApiLayer, Layer.mergeAll(DynamoLayer, TableLayer))

const Live = HttpRouter.serve(AppWithInfra, {
  middleware: requestLogger,
}).pipe(Layer.provide(HttpLive))
```

## Phase 3: Adding More Entities

Each new entity follows the same pattern: model → entity → filter → service → API → handler → wire into App.ts. We'll add Player (introducing shared PersonFields), Coach, Umpire, and Venue.

### 3.1 Shared person fields

Players, coaches, and umpires share common fields. Rather than inheritance, use object spread:

```ts
// src/models/Person.ts
import { Schema } from "effect"
import { GenderSchema } from "./Enums.js"

export const PersonFields = {
  firstName: Schema.String,
  lastName: Schema.String,
  name: Schema.String,
  dateOfBirth: Schema.optionalKey(Schema.Date),
  gender: GenderSchema,
}
```

> **Why composition over inheritance?** Each model gets its own copy of the fields. No base class to deal with — just plain objects spread into `Schema.Class`.

> **`Schema.optionalKey` vs `Schema.optional`** — `optionalKey` makes the property key itself optional (`{ dateOfBirth?: Date }`), meaning the key may be absent. `optional` makes the value optional (`{ dateOfBirth: Date | undefined }`), where the key is always present. With `exactOptionalPropertyTypes: true`, these are distinct types.

```ts
// Error helpers: catch domain errors, convert to HTTP errors
// ---------------------------------------------------------------------------

const entityNotFound =
  (entity: string, id: string) =>
  <A, E, R>(self: Effect.Effect<A, E | ItemNotFound, R>) =>
    Effect.catchTag(
      self,
      "ItemNotFound",
      () => Effect.fail(new ApiNotFound({ error: "NotFound", message: `${entity} not found: ${id}` })),
      (e) => Effect.die(e),
    )

const uniqueConflict =
  (entity: string) =>
  <A, E, R>(self: Effect.Effect<A, E | UniqueConstraintViolation, R>) =>
    Effect.catchTag(
      self,
      "UniqueConstraintViolation",
      () => Effect.fail(new ApiConflict({ error: "Conflict", message: `${entity} already exists (unique constraint violation)` })),
      (e) => Effect.die(e),
    )

// Handlers
// ---------------------------------------------------------------------------

const TeamsHandler = HttpApiBuilder.group(
```

## Phase 4: Entity References and Cascades

Now let's add SquadSelection — an entity that **references** other entities (Player and Team). When a Player or Team is updated, their denormalized data in SquadSelections should update too.

### 4.1 SquadSelection model


```ts region="model-squad-selection" example="cricket.ts"

// squadId encodes team + season + series (e.g., "aus#2024-25#BGT").

  squadId: Schema.String,
  selectionNumber: Schema.Number,
```

The `team` and `player` fields use `DynamoModel.ref` — this marks them as reference fields. In the **input** schema, these become `teamId` and `playerId` (you provide the ID). In the **record** schema, they contain the full `Team` and `Player` objects (hydrated on read).

### 4.2 SquadSelections entity with refs and cascade indexes

```ts region="entity-squad-selection" example="cricket.ts"
const SquadSelections = Entity.make({
  model: SquadSelection,
  entityType: "SquadSelection",
  indexes: {
    primary: {
      pk: { field: "pk", composite: ["squadId"] },
      sk: { field: "sk", composite: ["selectionNumber"] },
    },
    byPlayer: {
      index: "gsi1",
      pk: { field: "gsi1pk", composite: ["playerId"] },
      sk: { field: "gsi1sk", composite: ["squadId", "selectionNumber"] },
    },
  },
  refs: {
    team: { entity: Teams },
    player: { entity: Players },
  },
})
```

- **`refs`** — Maps ref fields to their source entities. Each ref field declared with `DynamoModel.ref` on the model is bound to its source entity here. The value `{ entity: Teams }` tells the library to hydrate the `team` field by looking up the referenced Team entity.

- **`byPlayer` GSI** — A secondary index keyed by `playerId`, allowing queries like "find all squad selections for a given player".

```ts
// src/services/SquadSelectionService.ts
import { DynamoClient, Entity } from "@effect-dynamodb/core"
import { type DateTime, Effect, ServiceMap } from "effect"
import { ulid } from "ulid"
import { SquadSelections } from "../entities/SquadSelections.js"
import type { SquadListFilter } from "../models/Filters.js"
import type { SquadSelectionId } from "../models/Ids.js"
import { applyPagination, type PaginationOptions } from "../models/Pagination.js"
import { MainTable } from "../entities/Schema.js"

export type { DateTime }

type CreateSquadSelectionInput = Entity.Create<typeof SquadSelections>

export class SquadSelectionService extends ServiceMap.Service<SquadSelectionService>()(
  "@gamemanager/SquadSelectionService",
  {
    make: Effect.gen(function* () {
      const { SquadSelections: squads } = yield* DynamoClient.make(MainTable)
      return {
        create: Effect.fn(function* (input: CreateSquadSelectionInput) {
          const id = ulid() as SquadSelectionId
          return yield* squads.put({ ...input, id })
        }),

        get: (id: SquadSelectionId) => squads.get({ id }),

        delete: (id: SquadSelectionId) => squads.delete({ id }),

        list: (filter: SquadListFilter = {}, pagination?: PaginationOptions) =>
          applyPagination(SquadSelections.query.byAll(filter), pagination).pipe(
            Query.execute,
            Effect.map((page) => ({
              data: page.items,
              count: page.items.length,
              cursor: page.cursor,
            })),
          ),
      }
    }),
  },
) {}
```

> **No update method** — SquadSelections are typically created and deleted, not updated. If you need to change a selection, delete and recreate it. The cascade mechanism handles updates to the *referenced* entities (Team, Player), not to the selection itself.

## Phase 5: The Match Aggregate

This is the most powerful feature of effect-dynamodb. A Match is a **graph of related entities** stored as multiple DynamoDB items but accessed as a single domain object.

### 5.1 Sub-aggregate models

A Match contains two TeamSheets (one per team). Each TeamSheet references a Team, a Coach, and an array of PlayerSheets:

```ts region="model-aggregate" example="cricket.ts"
// Aggregate domain schemas
class PlayerSheet extends Schema.Class<PlayerSheet>("PlayerSheet")({
  player: Player.pipe(DynamoModel.ref),
  battingPosition: Schema.Number,
  isCaptain: Schema.Boolean,
}) {}

class TeamSheet extends Schema.Class<TeamSheet>("TeamSheet")({
  team: Team.pipe(DynamoModel.ref),
  coach: Coach.pipe(DynamoModel.ref),
  homeTeam: Schema.Boolean,
  players: Schema.Array(PlayerSheet),
}) {}

class Match extends Schema.Class<Match>("Match")({
  id: Schema.String,
  name: Schema.String,
  venue: Venue.pipe(DynamoModel.ref),
  team1: TeamSheet,
  team2: TeamSheet,
}) {}
```

### 5.2 Match model with validation

```ts
// src/models/Match.ts
import { Schema } from "effect"
import { GenderSchema, MatchTypeSchema } from "./Enums.js"
import { Team } from "./Team.js"
import { TeamSheet } from "./TeamSheet.js"
import { UmpireSheet } from "./UmpireSheet.js"
import { Venue } from "./Venue.js"

export class Match extends Schema.Class<Match>("Match")({
  id: Schema.String,
  name: Schema.String,
  gender: GenderSchema,
  matchType: MatchTypeSchema,
  league: Schema.String,
  series: Schema.String,
  season: Schema.String,
  startDate: Schema.Date,
  finishDate: Schema.optionalKey(Schema.Date),
  venue: Venue,
  team1: TeamSheet,
  team2: TeamSheet,
  umpires: Schema.optionalKey(UmpireSheet),
  toss: Schema.optionalKey(Team),
  winner: Schema.optionalKey(Team),
  result: Schema.optionalKey(Schema.String),
}) {}

// Validation rules
const differentTeams = Schema.makeFilter<Match>((match) =>
  match.team1.team.id !== match.team2.team.id
    ? undefined
    : "Must have different teams in a match",
)

const minPlayersPerTeam = Schema.makeFilter<Match>((match) =>
  match.team1.players.length > 0 && match.team2.players.length > 0
    ? undefined
    : "Minimum of one player required per team",
)

export const ValidatedMatch = Match.check(differentTeams, minPlayersPerTeam)

/** Umpire assignment update — all fields optional, pass only what changes */
export const UmpireUpdate = Schema.Struct({
  matchRefereeId: Schema.optionalKey(Schema.String),
  matchUmpireIds: Schema.optionalKey(Schema.Array(Schema.String)),
  tvUmpireId: Schema.optionalKey(Schema.String),
  reserveUmpireId: Schema.optionalKey(Schema.String),
})

/** Partial update payload for matches — scalar fields + ref IDs resolved from match state */
export const MatchUpdatePayload = Schema.Struct({
  result: Schema.optionalKey(Schema.String),
  finishDate: Schema.optionalKey(Schema.Date),
  tossId: Schema.optionalKey(Schema.String),
  winnerId: Schema.optionalKey(Schema.String),
  umpires: Schema.optionalKey(UmpireUpdate),
})
```

The `MatchUpdatePayload` is a custom schema for the update endpoint. Aggregate updates operate on domain objects (`toss: Team`), but the API accepts ref IDs (`tossId: string`). The `UmpireUpdate` uses the same pattern — umpire role fields accept ID strings that the service resolves to full Umpire objects before the mutation (see section 5.5).

> **Concept: Schema.makeFilter** — Adds validation rules to a schema. Returns `undefined` for valid, or an error message string for invalid. `Match.check(...)` composes multiple filters into a validated schema.

### 5.3 The MatchAggregate

This is where the magic happens. The aggregate defines how a complex domain object decomposes into DynamoDB items and reassembles:

```ts region="aggregates" example="cricket.ts"
// Sub-aggregate: a team's composition within a match
const TeamSheetAggregate = Aggregate.make(TeamSheet, {
  root: { entityType: "MatchTeam" },
  edges: {
    team: Aggregate.ref(Teams),
    coach: Aggregate.one("coach", { entityType: "MatchCoach", entity: Coaches }),
    players: Aggregate.many("players", { entityType: "MatchPlayer", entity: Players }),
  },
})

// Top-level aggregate: the full match
const MatchAggregate = Aggregate.make(Match, {
  table: MainTable,
  schema: CricketSchema,
  pk: { field: "pk", composite: ["id"] },
  collection: {
    index: "lsi1",
    name: "match",
    sk: { field: "lsi1sk", composite: ["name"] },
  },
  root: { entityType: "MatchItem" },
  edges: {
    venue: Aggregate.one("venue", { entityType: "MatchVenue", entity: Venues }),
    team1: TeamSheetAggregate.with({ discriminator: { teamNumber: 1 } }),
    team2: TeamSheetAggregate.with({ discriminator: { teamNumber: 2 } }),
  },
})
```

Key aggregate concepts:

- **`Aggregate.ref(Teams)`** — Like `DynamoModel.ref` but at the aggregate level. The `team` field in `TeamSheet` references Teams. Input accepts IDs; output contains full Team objects.

- **`Aggregate.one("venue", { entity: Venues, entityType: "MatchVenue" })`** — A one-to-one edge. The venue is decomposed into a separate DynamoDB item with entity type `MatchVenue`, all sharing the match's partition key.

- **`Aggregate.many("players", { entity: Players, entityType: "MatchPlayer" })`** — A one-to-many edge. Each player in the team sheet becomes a separate DynamoDB item.

- **`TeamSheetAggregate.with({ discriminator: { teamNumber: 1 } })`** — Binds a sub-aggregate with a discriminator. Since a match has **two** TeamSheets, the discriminator (`teamNumber: 1` vs `teamNumber: 2`) distinguishes them in the sort key.

- **`collection`** — An index for fetching all items belonging to one match (the aggregate's "get" query). In this example it uses an LSI (`lsi1`).

### 5.5 MatchService with validation

Update the service to validate matches after create and update using the `ValidatedMatch` schema from section 5.2:

```ts
// src/services/MatchService.ts
import { Aggregate, DynamoClient } from "@effect-dynamodb/core"
import { type DateTime, Effect, Schema, ServiceMap } from "effect"
import { ulid } from "ulid"
import { MatchAggregate } from "../entities/MatchAggregate.js"
import type { UmpireId } from "../models/Ids.js"
import { type MatchUpdatePayload, ValidatedMatch } from "../models/Match.js"
import type { PaginationOptions } from "../models/Pagination.js"
import { MainTable } from "../entities/Schema.js"

export type { DateTime }

type MatchKey = Aggregate.Key<typeof MatchAggregate>
type MatchCreate = Schema.Schema.Type<typeof MatchAggregate.createSchema>

const validate = (match: Aggregate.Type<typeof MatchAggregate>) =>
  Schema.decodeUnknownEffect(ValidatedMatch)(match)

export class MatchService extends ServiceMap.Service<MatchService>()("@gamemanager/MatchService", {
  make: Effect.gen(function* () {
    const { Umpires: boundUmpires, MatchAggregate: matches } = yield* DynamoClient.make(MainTable)
    return {
      create: Effect.fn(function* (input: MatchCreate) {
        const id = ulid()
        const match = yield* matches.create({ ...input, id })
        yield* validate(match)
        return match
      }),

      get: (key: MatchKey) => matches.get(key),

      update: (key: MatchKey, payload: typeof MatchUpdatePayload.Type) =>
        Effect.gen(function* () {
          // Resolve umpire IDs to full objects before the mutation
          const umpireUpdates = payload.umpires
          const resolvedUmpires = umpireUpdates
            ? yield* Effect.gen(function* () {
                const matchReferee = umpireUpdates.matchRefereeId
                  ? yield* boundUmpires.get({ id: umpireUpdates.matchRefereeId as UmpireId })
                  : undefined
                const matchUmpire = umpireUpdates.matchUmpireIds
                  ? yield* Effect.all(
                      umpireUpdates.matchUmpireIds.map((id) =>
                        boundUmpires.get({ id: id as UmpireId }),
                      ),
                    )
                  : undefined
                const tvUmpire = umpireUpdates.tvUmpireId
                  ? yield* boundUmpires.get({ id: umpireUpdates.tvUmpireId as UmpireId })
                  : undefined
                const reserveUmpire = umpireUpdates.reserveUmpireId
                  ? yield* boundUmpires.get({ id: umpireUpdates.reserveUmpireId as UmpireId })
                  : undefined
                return { matchReferee, matchUmpire, tvUmpire, reserveUmpire }
              })
            : undefined

        const match = yield* matches.update(key, ({ state }) => {
          const patched = { ...state }
          if (payload.result !== undefined) patched.result = payload.result
          if (payload.finishDate !== undefined) patched.finishDate = payload.finishDate
          if (payload.tossId !== undefined) {
            if (state.team1.team.id === payload.tossId) patched.toss = state.team1.team
            else if (state.team2.team.id === payload.tossId) patched.toss = state.team2.team
          }
          if (payload.winnerId !== undefined) {
            if (state.team1.team.id === payload.winnerId) patched.winner = state.team1.team
            else if (state.team2.team.id === payload.winnerId) patched.winner = state.team2.team
          }
          if (resolvedUmpires) {
            const currentUmpires = state.umpires ?? {}
            patched.umpires = {
              ...currentUmpires,
              ...(resolvedUmpires.matchReferee !== undefined && {
                matchReferee: resolvedUmpires.matchReferee,
              }),
              ...(resolvedUmpires.matchUmpire !== undefined && {
                matchUmpire: resolvedUmpires.matchUmpire,
              }),
              ...(resolvedUmpires.tvUmpire !== undefined && {
                tvUmpire: resolvedUmpires.tvUmpire,
              }),
              ...(resolvedUmpires.reserveUmpire !== undefined && {
                reserveUmpire: resolvedUmpires.reserveUmpire,
              }),
            }
          }
          return patched
        })
        yield* validate(match)
        return match
      }),

      delete: (key: MatchKey) => matches.delete(key),

      list: (filter?: { gender?: string; league?: string }, pagination?: PaginationOptions) =>
        matches.list(
          filter?.gender
            ? { gender: filter.gender, ...(filter.league ? { league: filter.league } : {}) }
            : undefined,
          { limit: pagination?.count, cursor: pagination?.cursor },
        ).pipe(
          Effect.map((result) => ({
            ...result,
            count: result.data.length,
          })),
        ),
    }
  }),
}) {}
```

Key differences from entity services:

- **`DynamoClient.make(MainTable)`** — Both entities and aggregates are registered on the table and bound via the same typed gateway. Destructure `Umpires` (bound entity) and `MatchAggregate` (bound aggregate) from a single call — all operations have `R = never`.

- **`matches.createSchema`** — Like entities, aggregates derive typed schemas. `createSchema` omits PK composites and replaces ref/edge fields with ID fields (`venueId`, `team1.teamId`).

- **`matches.create(...)`** — Decomposes the input into multiple DynamoDB items and writes them atomically in a transaction.

- **`matches.get(key)`** — Queries all items in the aggregate's partition, then assembles them back into the full Match domain object.

- **`matches.update(key, fn)`** — The update function receives `{ state }` (the current Match as a plain object) and returns the modified state. The library diffs the old and new states, determines which items changed, and writes only the changes atomically.

- **Ref resolution from state** — The `tossId` and `winnerId` in the API payload are ref IDs, but the aggregate mutation operates on domain objects. The service resolves these by matching against the match's existing `team1.team` and `team2.team` — the toss/winner must be one of the two playing teams, so no additional DB lookup is needed.

- **Umpire resolution via entity lookup** — Unlike toss/winner (which can be resolved from the match's existing state), umpire updates reference umpires by ID that may not be in the current match state. The service resolves these by calling `boundUmpires.get()` *before* the mutation function. If an umpire ID doesn't exist, `ItemNotFound` is raised and caught as a 422 in the handler.

- **`MatchAggregate.list(filter, options)`** — Supports cursor-based pagination via `Aggregate.ListOptions`. Returns `Aggregate.ListResult<Match>` with `{ data, cursor }`.

- **`validate`** — Decodes the assembled `Match` through `ValidatedMatch`, which runs the `differentTeams` and `minPlayersPerTeam` checks. If validation fails, the schema decode produces a `SchemaError`. The `validationFailed` error helper in `App.ts` catches this and returns a `400 Bad Request` with the validation message. In a production system you'd validate *before* writing, but this pattern demonstrates the concept clearly.

> **Concept: Post-assembly validation** — The aggregate handles decomposition, storage, and assembly. Validation is a service concern applied to the assembled domain object. This keeps the aggregate focused on data mapping and the service focused on business rules.

```ts
// In App.ts — add error helper (after refNotFound):
const aggregateNotFound =
  (entity: string, id: string) =>
  <A, E, R>(self: Effect.Effect<A, E | AggregateAssemblyError, R>) =>
    Effect.catchTag(
      self,
      "AggregateAssemblyError",

      (e) => Effect.die(e),
    )
```

```ts
// In App.ts — add handler (before HandlerLayers):
const MatchesHandler = HttpApiBuilder.group(
  GameManagerApi,
  "matches",
  Effect.fn(function* (handlers) {
    const svc = yield* MatchService
    return handlers
      .handle("list", ({ query }) =>
        Effect.gen(function* () {
          const filter = query.gender ? { gender: query.gender, league: query.league } : undefined
          const result = yield* svc.list(filter, query)
          return {
            data: result.data.map((r) => new Match(r)),
            count: result.count,
            cursor: result.cursor,
          }
        }).pipe(Effect.orDie),
      )
      .handle("get", ({ params: { id } }) =>
        svc.get({ id }).pipe(aggregateNotFound("Match", id)),
      )
      .handle("create", ({ payload }) =>
        svc.create(payload).pipe(
          Effect.catchTags(
            {
              SchemaError: (e) =>
                Effect.fail(
                  new ApiBadRequest({ error: "BadRequest", message: String(e).replace(/^SchemaError\(/, "").replace(/\)$/, "") }),
                ),
              RefNotFound: (caught) => {
                const e = caught as RefNotFound
                return Effect.fail(
                  new ApiUnprocessableEntity({ error: "UnprocessableEntity", message: `${e.refEntity} not found: ${e.refId}` }),
                )
              },
            },
            (e) => Effect.die(e),
          ),
        ),
      )
      .handle("update", ({ params: { id }, payload }) =>
        svc.update({ id }, payload).pipe(
          Effect.catchTags(
            {
              SchemaError: (e) =>
                Effect.fail(
                  new ApiBadRequest({ error: "BadRequest", message: String(e).replace(/^SchemaError\(/, "").replace(/\)$/, "") }),
                ),
              AggregateAssemblyError: () =>
                Effect.fail(
                  new ApiNotFound({ error: "NotFound", message: `Match not found: ${id}` }),
                ),
              ItemNotFound: (e) =>
                Effect.fail(
                  new ApiUnprocessableEntity({ error: "UnprocessableEntity", message: `Umpire not found: ${JSON.stringify((e as ItemNotFound).key)}` }),
                ),
            },
            (e) => Effect.die(e),
          ),
        ),
      )
      .handle("remove", ({ params: { id } }) =>
        svc.delete({ id }).pipe(aggregateNotFound("Match", id)),
      )
  }),
)
```

Key differences from entity handlers:

- **`svc.get({ id })`** — Aggregate keys are objects (not scalar IDs), matching the PK composite structure.
- **`svc.update({ id }, payload)`** — Passes the `MatchUpdatePayload` directly to the service. The service resolves toss/winner from the match's existing teams, and umpire IDs via entity lookup (see section 5.5).
- **`aggregateNotFound`** — Catches `AggregateAssemblyError` instead of `ItemNotFound`.
- **`ItemNotFound` catch** — When the service resolves an umpire ID that doesn't exist, the `Umpires.get()` call raises `ItemNotFound`. The handler catches this and returns a `422 Unprocessable Entity` with the failed key.
- **`Effect.catchTags`** — When a handler can produce multiple error types, use `Effect.catchTags` to handle them all in one combinator. This avoids the pitfall of piping individual `Effect.catchTag` helpers — each helper's fallback `(e) => Effect.die(e)` converts ALL unmatched errors to defects, preventing downstream handlers from catching them. `Effect.catchTags` handles all specified tags atomically, with a single fallback for everything else.

```bash
  -H "Content-Type: application/json" \
  -d "{\"umpires\": {\"tvUmpireId\": \"$UMP2\", \"matchRefereeId\": \"$UMP1\"}}" \
  | jq '{matchReferee: .umpires.matchReferee.name, tvUmpire: .umpires.tvUmpire.name}'
# → {"matchReferee": "Kumar Dharmasena", "tvUmpire": "Richard Kettleborough"}

# Try an invalid umpire ID — should get 422
curl -s -X PUT http://localhost:3000/matches/$MATCH_ID \

```

Umpire updates demonstrate cascading updates within an aggregate. The service resolves umpire IDs to full objects before the mutation, then the aggregate engine diffs old vs new and rewrites only the changed umpire sub-aggregate items — the team sheets and venue items are untouched.

## Concept Summary

| Concept | What it does | Where you used it |
|---------|-------------|-------------------|
| **Schema.Class** | Runtime-validated domain model | Every model file |
| **Schema.brand** | Branded nominal types for IDs — preserved through Entity.Input, Entity.Key, and ref types | Ids.ts, services, API params |
| **Schema.optionalKey** | Optional property key (may be absent) | All optional fields |
| **DynamoModel.configure** | Rename fields, mark identifier/ref at entity level | Every entity |
| **DynamoSchema + Table** | Application namespace + physical table binding | Schema.ts |
| **Entity.make** | Binds model to table with indexes | Every entity |
| **Entity.createSchema** | Derived schema: input fields minus PK composites, ref-aware | API create payloads |
| **Entity.updateSchema** | Derived schema: optional non-key fields, ref-aware | API update payloads |
| **Entity.cascade** | Propagate updates to referencing entities | TeamService, PlayerService |
| **Query combinators** | Composable query building | Pagination helpers, services |
| **ServiceMap.Service** | Dependency-injectable service | Every service |
| **Aggregate.make** | Graph-based composite entity | MatchAggregate |
| **Aggregate.createSchema** | Derived schema: input minus PK, edges→IDs | MatchService create |
| **Aggregate.updateSchema** | Derived schema: all input fields optional | MatchService update |
| **Aggregate.list** | Cursor-based paginated listing | MatchService list |
| **Aggregate.ListOptions / ListResult** | Pagination input/output types | MatchService list |
| **Aggregate.ref** | Inline reference hydration | Match toss/winner |
| **Aggregate.one / many** | Decompose to separate items | Venue, Coach, Players |
| **Sub-aggregate + discriminator** | Reuse aggregate structure with differentiation | team1/team2 TeamSheets |
| **Schema.makeFilter + check** | Composable validation rules on domain models | Match differentTeams, minPlayers |
| **Table.definition (aggregates)** | Include aggregate GSIs in table creation | create-table.ts |
| **AggregateAssemblyError** | Error when aggregate partition has no items | MatchesHandler not-found |
| **Schema.ErrorClass API errors** | Typed HTTP errors with constructors, status codes, clean JSON | ApiErrors.ts |
| **Effect.catchTag** | Convert domain errors to HTTP errors | App.ts error helpers |
| **validationFailed** | Catch SchemaError from `.check()`, return 400 | Match create/update handlers |
| **Layer composition** | Wire everything together at runtime | App.ts, server.ts |
