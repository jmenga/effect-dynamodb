A shopping mall directory application managing **mall stores** in a single DynamoDB table. Adapted from the [ElectroDB Shopping Mall example](https://electrodb.dev/en/examples/shopping-mall/) to showcase effect-dynamodb's approach to the same problem.

**What you'll learn:**
- Single entity with a composite primary key (2 PK + 2 SK composites)
- 2 GSIs for different access patterns on the same entity
- Sort key prefix queries with `beginsWith` to filter by building
- Date range queries with `between` for lease expiration windows
- CRUD operations with composite keys
- Layer-based dependency injection

---

## Access Patterns

This example supports six access patterns on a single entity:

1. **Get store** — Look up a specific store by city, mall, building, and store ID (primary key)
2. **Stores in a mall** — Find all stores in a given mall (GSI1)
3. **Stores in a building** — Narrow to a specific building within a mall (GSI1 + `beginsWith`)
4. **Lease renewals** — Find leases expiring within a date range (GSI2 + `between`)
5. **Update store** — Change rent and discount for a store
6. **Delete store** — Remove a store from the directory

---

## Step 1: Model

A pure domain model with no DynamoDB concepts:

```typescript title="models.ts"
import { Schema } from "effect"

class MallStore extends Schema.Class<MallStore>("MallStore")({
  cityId: Schema.String,
  mallId: Schema.String,
  storeId: Schema.String,
  buildingId: Schema.String,
  unitId: Schema.String,
  category: Schema.Literals([
    "food/coffee",
    "food/meal",
    "clothing",
    "electronics",
    "department",
    "misc",
  ]),
  leaseEndDate: Schema.String,
  rent: Schema.String,
  discount: Schema.String.pipe(Schema.withDecodingDefaultKey(() => "0.00")),
  deposit: Schema.optional(Schema.Number),
}) {}
```

Key modeling decisions:
- **`category`** uses `Schema.Literals` for a constrained set of store types
- **`discount`** has a decoding default of `"0.00"` — if omitted during `put`, it defaults automatically
- **`deposit`** is optional — not every store has one
- **`rent`**, **`leaseEndDate`** are strings for lexicographic sort key ordering in GSI queries

## Step 2: Schema, Table, and Entity

### Schema and Table

```typescript title="entities.ts"
import { DynamoSchema, Table } from "effect-dynamodb"

const MallSchema = DynamoSchema.make({ name: "mall", version: 1 })
const MallTable = Table.make({ schema: MallSchema })
```

### MallStore Entity

The entity defines three indexes — a composite primary key plus two GSIs:

```typescript title="entities.ts"
import { Entity } from "effect-dynamodb"

const MallStores = Entity.make({
  model: MallStore,
  entityType: "MallStore",
  indexes: {
    primary: {
      pk: { field: "pk", composite: ["cityId", "mallId"] },
      sk: { field: "sk", composite: ["buildingId", "storeId"] },
    },
    units: {
      index: "gsi1",
      pk: { field: "gsi1pk", composite: ["mallId"] },
      sk: { field: "gsi1sk", composite: ["buildingId", "unitId"] },
    },
    leases: {
      index: "gsi2",
      pk: { field: "gsi2pk", composite: ["storeId"] },
      sk: { field: "gsi2sk", composite: ["leaseEndDate"] },
    },
  },
  timestamps: true,
})
```

| Index | PK composites | SK composites | Access pattern |
|-------|--------------|---------------|----------------|
| **primary** | `cityId`, `mallId` | `buildingId`, `storeId` | Get a specific store |
| **units** (GSI1) | `mallId` | `buildingId`, `unitId` | Stores in a mall or building |
| **leases** (GSI2) | `storeId` | `leaseEndDate` | Lease expiration queries |

The composite primary key uses four attributes — `cityId` and `mallId` form the partition key, while `buildingId` and `storeId` form the sort key. This means a `get` requires all four values.

## Step 4: Access Patterns

### Pattern 1: Get Store by Primary Key

The composite primary key requires all four attributes — `cityId`, `mallId`, `buildingId`, and `storeId`:

```typescript
const coffee = yield* mallStores.get({
  cityId: "atlanta",
  mallId: "eastpointe",
  buildingId: "bldg-a",
  storeId: "star-coffee",
})
// coffee.category → "food/coffee"
// coffee.rent → "3500.00"
// coffee.deposit → 7000
```

### Pattern 2: All Stores in a Mall (GSI1)

The `units` index on GSI1 uses `mallId` as the partition key. Querying with just the PK composites returns all stores in that mall:

```typescript
import { Query } from "effect-dynamodb"

const eastpointeStores = yield* mallStores.collect(
  mallStores.query.units({ mallId: "eastpointe" }),
)
// → 4 stores: star-coffee, burger-barn, tech-zone, trendy-threads

const westgateStores = yield* mallStores.collect(
  mallStores.query.units({ mallId: "westgate" }),
)
// → 2 stores: mega-mart, gifts-galore
```

### Pattern 3: Stores in a Building (GSI1 + beginsWith)

To narrow to a specific building, compose a sort key prefix with `KeyComposer.composeSortKeyPrefix` and use `Query.where({ beginsWith })`:

```typescript
import { KeyComposer, Query } from "effect-dynamodb"

const unitsIndex = MallStores.indexes.units
const bldgAPrefix = KeyComposer.composeSortKeyPrefix(
  MallSchema, "MallStore", 1, unitsIndex, { buildingId: "bldg-a" },
)

const bldgAStores = yield* mallStores.collect(
  mallStores.query.units({ mallId: "eastpointe" }).pipe(
    Query.where({ beginsWith: bldgAPrefix }),
  ),
)
// → 2 stores: star-coffee, burger-barn

const bldgBPrefix = KeyComposer.composeSortKeyPrefix(
  MallSchema, "MallStore", 1, unitsIndex, { buildingId: "bldg-b" },
)

const bldgBStores = yield* mallStores.collect(
  mallStores.query.units({ mallId: "eastpointe" }).pipe(
    Query.where({ beginsWith: bldgBPrefix }),
  ),
)
// → 2 stores: tech-zone, trendy-threads
```

The sort key for `units` is composed from `buildingId + unitId`. By composing a prefix from just `buildingId`, the `beginsWith` condition matches all stores in that building regardless of `unitId`.

### Pattern 4: Lease Renewals by Date Range (GSI2)

The `leases` index on GSI2 uses `storeId` as the partition key and `leaseEndDate` as the sort key. Compose sort key bounds to query a date range with `between`:

```typescript
import { KeyComposer, Query } from "effect-dynamodb"

const leasesIndex = MallStores.indexes.leases

// Q1 2025 bounds
const q1Lo = KeyComposer.composeSortKeyPrefix(
  MallSchema, "MallStore", 1, leasesIndex, { leaseEndDate: "2025-01-01" },
)
const q1Hi = KeyComposer.composeSortKeyPrefix(
  MallSchema, "MallStore", 1, leasesIndex, { leaseEndDate: "2025-03-31" },
)

// star-coffee lease ends 2025-03-31 — within Q1
const starCoffeeQ1 = yield* mallStores.collect(
  mallStores.query.leases({ storeId: "star-coffee" }).pipe(
    Query.where({ between: [q1Lo, q1Hi] }),
  ),
)
// → 1 result (lease ends 2025-03-31)

// tech-zone lease ends 2025-09-30 — NOT in Q1
const techZoneQ1 = yield* mallStores.collect(
  mallStores.query.leases({ storeId: "tech-zone" }).pipe(
    Query.where({ between: [q1Lo, q1Hi] }),
  ),
)
// → 0 results

// Q3 2025 bounds for tech-zone
const q3Lo = KeyComposer.composeSortKeyPrefix(
  MallSchema, "MallStore", 1, leasesIndex, { leaseEndDate: "2025-07-01" },
)
const q3Hi = KeyComposer.composeSortKeyPrefix(
  MallSchema, "MallStore", 1, leasesIndex, { leaseEndDate: "2025-09-30" },
)

const techZoneQ3 = yield* mallStores.collect(
  mallStores.query.leases({ storeId: "tech-zone" }).pipe(
    Query.where({ between: [q3Lo, q3Hi] }),
  ),
)
// → 1 result (lease ends 2025-09-30)
```

Because `leaseEndDate` is an ISO date string, it sorts lexicographically in chronological order -- no padding or transformation needed.

### Pattern 5: Update Store

Update non-key attributes with `set()`. The composite key identifies which item to update:

```typescript
const updated = yield* mallStores.update({
  cityId: "atlanta",
  mallId: "eastpointe",
  buildingId: "bldg-a",
  storeId: "star-coffee",
}, {
  rent: "4000.00",
  discount: "100.00",
})
// updated.rent → "4000.00"
// updated.discount → "100.00"
// updated.category → "food/coffee" (preserved)
```

### Pattern 6: Delete Store

Delete requires the full composite key:

```typescript
yield* mallStores.delete({
  cityId: "atlanta",
  mallId: "westgate",
  buildingId: "bldg-c",
  storeId: "gifts-galore",
})

// Verify: westgate now has 1 store
const westgateAfter = yield* mallStores.collect(
  mallStores.query.units({ mallId: "westgate" }),
)
// → 1 store: mega-mart
```

After deletion, the store disappears from both the primary key and all GSI queries.

### Layer Setup

The example wires dependencies via Effect Layers -- no global config or constructor options:

```typescript title="main.ts"
import { Effect, Layer } from "effect"
import { DynamoClient } from "effect-dynamodb"

const AppLayer = Layer.mergeAll(
  DynamoClient.layer({
    region: "us-east-1",
    endpoint: "http://localhost:8000",
    credentials: { accessKeyId: "local", secretAccessKey: "local" },
  }),
  MallTable.layer({ name: "mall-table" }),
)

const main = program.pipe(Effect.provide(AppLayer))

Effect.runPromise(main).then(
  () => console.log("Done."),
  (err) => console.error("Failed:", err),
)
```

## Key Takeaways

| Concept | How it's used |
|---------|---------------|
| **Composite primary key** | 4 attributes (2 PK + 2 SK) uniquely identify each store |
| **GSI for alternate access** | GSI1 (`units`) queries by mall; GSI2 (`leases`) queries by store + date |
| **Sort key prefix queries** | `beginsWith` on `buildingId` filters stores within a building |
| **Date range queries** | `between` on `leaseEndDate` finds leases expiring in a date window |
| **Schema defaults** | `discount` defaults to `"0.00"` via `Schema.withDecodingDefaultKey` |
| **Optional fields** | `deposit` is `Schema.optional` -- not every store requires one |
