A comprehensive example demonstrating the full effect-dynamodb high-level API. Starting from model definitions through to advanced lifecycle features, this example covers every major operation you will use when building applications.

**What you'll learn:**
- Creating items with `put` and reading with `get`
- Updates with variadic combinators: `update(key, Entity.set(...), Entity.expectedVersion(n))`
- Error handling with tagged errors (`ItemNotFound`, `OptimisticLockError`)
- GSI queries with combinators (`where`, `reverse`, `collect`)
- Atomic transactions with `Transaction.transactWrite` and `transactGet`
- Sparse GSI behavior and the all-or-none update constraint
- Version history with `versioned: { retain: true }`
- Soft delete, restore, and purge

---

## Step 1: Define Models

Pure domain models with no DynamoDB concepts:

```typescript title="models.ts"
import { Schema } from "effect"
import { DynamoModel } from "effect-dynamodb"

class User extends Schema.Class<User>("User")({
  userId: Schema.String,
  email: Schema.String,
  displayName: Schema.NonEmptyString,
  role: Schema.Literals(["admin", "member"]),
  createdBy: Schema.String,
}) {}

const UserModel = DynamoModel.configure(User, {
  createdBy: { immutable: true },
})

class Task extends Schema.Class<Task>("Task")({
  taskId: Schema.String,
  userId: Schema.String,
  title: Schema.NonEmptyString,
  status: Schema.Literals(["todo", "in-progress", "done"]),
  priority: Schema.Number,
}) {}
```

`createdBy` is marked `immutable` -- once set during creation, updates cannot change it.

## Step 2: Schema, Table, and Entities

```typescript title="entities.ts"
import { DynamoSchema, Entity, Table } from "effect-dynamodb"

const AppSchema = DynamoSchema.make({ name: "crud-demo", version: 1 })
const Users = Entity.make({
  model: UserModel,
  entityType: "User",
  indexes: {
    primary: {
      pk: { field: "pk", composite: ["userId"] },
      sk: { field: "sk", composite: [] },
    },
    byRole: {
      index: "gsi1",
      pk: { field: "gsi1pk", composite: ["role"] },
      sk: { field: "gsi1sk", composite: ["userId"] },
    },
  },
  timestamps: true,
  versioned: { retain: true },
})

const Tasks = Entity.make({
  model: Task,
  entityType: "Task",
  indexes: {
    primary: {
      pk: { field: "pk", composite: ["taskId"] },
      sk: { field: "sk", composite: [] },
    },
    byUser: {
      index: "gsi1",
      pk: { field: "gsi1pk", composite: ["userId"] },
      sk: { field: "gsi1sk", composite: ["status", "taskId"] },
    },
  },
  timestamps: true,
  versioned: true,
  softDelete: true,
})

const MainTable = Table.make({ schema: AppSchema, entities: { Users, Tasks } })
```

Key configuration:
- **Users** have `versioned: { retain: true }` -- every mutation creates a version snapshot in the same partition, enabling full audit history
- **Users** have a `byRole` GSI for querying users by their role
- **Tasks** have `softDelete: true` -- deleting archives the item rather than destroying it
- **Tasks** have `status` and `taskId` in the sort key composite, so tasks for a user are sorted by status

## Step 3: Typed Client and Create Items

Before calling any entity operations, obtain a typed client via `DynamoClient.make(MainTable)` to resolve the `DynamoClient` and `TableConfig` dependencies. The returned client provides bound entities where all operations have `R = never`:

```typescript title="put.ts"
import { DynamoClient } from "effect-dynamodb"

const { Users: users, Tasks: tasks } = yield* DynamoClient.make(MainTable)

// Returns the model type (User)
const alice = yield* users.put({
  userId: "u-alice",
  email: "alice@example.com",
  displayName: "Alice",
  role: "admin",
  createdBy: "system",
})
// alice.userId, alice.email, alice.displayName, alice.role, alice.createdBy

const bob = yield* users.put({
  userId: "u-bob",
  email: "bob@example.com",
  displayName: "Bob",
  role: "member",
  createdBy: "system",
})
```

The same pattern applies to Tasks:

```typescript title="put.ts"
yield* tasks.put({
  taskId: "t-1",
  userId: "u-alice",
  title: "Design the API",
  status: "done",
  priority: 1,
})

yield* tasks.put({
  taskId: "t-2",
  userId: "u-alice",
  title: "Write tests",
  status: "in-progress",
  priority: 2,
})
```

## Step 4: Reading Items

`get` returns the model type by default:

```typescript title="get.ts"
// Returns the clean domain type (User)
const user = yield* users.get({ userId: "u-alice" })
// user.userId, user.displayName, user.role
```

## Step 5: Updates with Variadic Combinators

`update(key, ...combinators)` accepts `Entity.set()`, `Entity.expectedVersion()`, and other combinators as variadic rest arguments:

```typescript title="update.ts"
// Simple field update
const updatedAlice = yield* users.update(
  { userId: "u-alice" },
  Entity.set({ role: "member", displayName: "Alice W." }),
)
// updatedAlice: User -- clean model type

// Multiple combinators composed together
const updatedAgain = yield* users.update(
  { userId: "u-alice" },
  Entity.set({ displayName: "Alice Wu" }),
)
// updatedAgain.createdBy → "system" (immutable -- unchanged)
```

---

## Step 6: Optimistic Locking

`Entity.expectedVersion()` composes with `Entity.set()` as variadic combinators. If the item's current version does not match, the update fails with `OptimisticLockError`:

```typescript title="locking.ts"
import { Effect } from "effect"

const result = yield* users.update(
  { userId: "u-alice" },
  Entity.set({ displayName: "Alice Wrong" }),
  Entity.expectedVersion(1), // item is at version 3 -- this will fail
).pipe(
  Effect.catchTag("OptimisticLockError", (e) =>
    Effect.succeed(`Expected v${e.expectedVersion}, item has been updated`),
  ),
)
```

## Step 7: Error Handling

All errors are tagged, enabling precise `catchTag` handling. `ItemNotFound` is the most common:

```typescript title="errors.ts"
const result = yield* users.get({ userId: "u-nonexistent" })
  .pipe(
    Effect.map((u) => `Found: ${u.userId}`),
    Effect.catchTag("ItemNotFound", (e) =>
      Effect.succeed(`Not found: entity=${e.entityType}, key=${JSON.stringify(e.key)}`),
    ),
  )
```

Bound entity methods return `Effect` directly, so you can pipe to Effect combinators like `Effect.map` or `Effect.catchTag` without any conversion.

## Step 8: GSI Queries

Query a GSI by providing the partition key composites. Use `Query.collect` to gather all results:

```typescript title="queries.ts"
// All tasks for a user
const aliceTasks = yield* tasks.collect(Tasks.query.byUser({ userId: "u-alice" }))

// All admins
const admins = yield* users.collect(Users.query.byRole({ role: "admin" }))
```

### Query Combinators

Compose query behavior with pipeable combinators:

```typescript title="queries.ts"
import { Query } from "effect-dynamodb"

// Reverse sort order
const reversed = yield* Tasks.query
  .byUser({ userId: "u-alice" })
  .pipe(Query.reverse, (q) => tasks.collect(q))

// Sort key prefix filter (beginsWith)
const todoTasks = yield* Tasks.query
  .byUser({ userId: "u-alice" })
  .pipe(Query.where({ beginsWith: "$crud-demo#v1#task#todo" }), (q) => tasks.collect(q))
```

`Query.reverse` flips the scan direction. `Query.where` adds a sort key condition -- here, `beginsWith` filters to tasks with a `todo` status prefix in the sort key.

## Step 9: Atomic Transactions

### transactWrite

Atomically create multiple items across entities. If either fails, neither is persisted:

```typescript title="transactions.ts"
import { Transaction } from "effect-dynamodb"

yield* Transaction.transactWrite([
  Users.put({
    userId: "u-charlie",
    email: "charlie@example.com",
    displayName: "Charlie",
    role: "member",
    createdBy: "admin",
  }),
  Tasks.put({
    taskId: "t-5",
    userId: "u-charlie",
    title: "Onboarding checklist",
    status: "todo",
    priority: 1,
  }),
])
```

### transactGet

Fetch multiple items atomically with a typed tuple return:

```typescript title="transactions.ts"
const [txAlice, txBob, txTask] = yield* Transaction.transactGet([
  Users.get({ userId: "u-alice" }),
  Users.get({ userId: "u-bob" }),
  Tasks.get({ taskId: "t-1" }),
])
// txAlice: User | undefined
// txBob: User | undefined
// txTask: Task | undefined
```

Each element in the returned tuple is typed to its entity's model, or `undefined` if the item was not found.

## Step 10: Sparse GSI Behavior

When GSI composite attributes are optional, DynamoDB's sparse index behavior applies -- items only appear in the index when their key attributes are present:

```typescript title="sparse.ts"
class Project extends Schema.Class<Project>("Project")({
  projectId: Schema.String,
  name: Schema.NonEmptyString,
  ownerId: Schema.optional(Schema.String),
  department: Schema.optional(Schema.String),
}) {}

const Projects = Entity.make({
  model: Project,
  entityType: "Project",
  indexes: {
    primary: {
      pk: { field: "pk", composite: ["projectId"] },
      sk: { field: "sk", composite: [] },
    },
    byOwner: {
      index: "gsi1",
      pk: { field: "gsi1pk", composite: ["ownerId"] },
      sk: { field: "gsi1sk", composite: ["department"] },
    },
  },
  timestamps: true,
})

// assumes MainTable is defined with entities: { Projects }
const { Projects: projects } = yield* DynamoClient.make(MainTable)

// Assigned project -- appears in byOwner GSI
yield* projects.put({
  projectId: "p-1",
  name: "Alpha",
  ownerId: "u-alice",
  department: "engineering",
})

// Unassigned project -- does NOT appear in byOwner GSI (sparse)
yield* projects.put({
  projectId: "p-2",
  name: "Beta",
  // ownerId and department omitted
})

// Query only returns assigned projects
const ownerProjects = yield* projects.collect(
  Projects.query.byOwner({ ownerId: "u-alice" }),
)
// → [{ projectId: "p-1", name: "Alpha" }]
// "Beta" is absent -- sparse index at work
```

### All-or-None Update Constraint

When updating, if the payload includes ANY composite for a GSI, ALL composites for that GSI must be present. This prevents stale GSI entries:

```typescript title="sparse.ts"
// assumes MainTable is defined with entities: { Employees }
const { Employees: employees } = yield* DynamoClient.make(MainTable)

// GOOD: Update without touching GSI composites
yield* employees.update({ employeeId: "e-1" }, Entity.set({ name: "Alice W." }))

// GOOD: Update ALL composites of a GSI
yield* employees.update({ employeeId: "e-1" }, Entity.set({ tenantId: "t-2", region: "eu-west-1" }))

// BAD: Partial GSI composites -- runtime ValidationError
yield* employees.update(
  { employeeId: "e-1" },
  Entity.set({ tenantId: "t-3" }), // missing region!
).pipe(
  Effect.catchTag("ValidationError", (e) =>
    Effect.succeed(`Must provide both tenantId AND region, or neither`),
  ),
)
```

## Step 11: Hard Delete

For entities without `softDelete`, `Entity.delete` permanently removes the item:

```typescript title="delete.ts"
yield* users.delete({ userId: "u-charlie" })

// Confirm deletion
const result = yield* users.get({ userId: "u-charlie" })
  .pipe(
    Effect.map(() => "still exists!"),
    Effect.catchTag("ItemNotFound", () => Effect.succeed("confirmed deleted")),
  )
```

## Step 12: Version History

With `versioned: { retain: true }`, every put and update creates a snapshot in the same DynamoDB partition. Retrieve individual versions or query the full history:

```typescript title="versions.ts"
// Fetch a specific version
const aliceV1 = yield* users.getVersion({ userId: "u-alice" }, 1)
// aliceV1.displayName → "Alice", aliceV1.role → "admin"

const aliceV2 = yield* users.getVersion({ userId: "u-alice" }, 2)
// aliceV2.displayName → "Alice W.", aliceV2.role → "member"

// Query all version snapshots
const allVersions = yield* users.collect(Users.versions({ userId: "u-alice" }))
// → [{ version: 1, ... }, { version: 2, ... }, { version: 3, ... }]

// Non-existent version → ItemNotFound
yield* users.getVersion({ userId: "u-alice" }, 99)
  .pipe(Effect.catchTag("ItemNotFound", () => Effect.succeed("not found")))
```

Version snapshots share the primary partition key but use a version-specific sort key. GSI keys are stripped from snapshots so they do not appear in index queries.

## Step 13: Soft Delete, Restore, and Purge

### Soft Delete

For entities with `softDelete: true`, `Entity.delete` archives the item instead of destroying it. GSI keys are stripped, so the item vanishes from all index queries:

```typescript title="soft-delete.ts"
yield* tasks.delete({ taskId: "t-1" })

// No longer in GSI queries
const activeTasks = yield* tasks.collect(Tasks.query.byUser({ userId: "u-alice" }))
// t-1 is absent

// But retrievable via deleted.get
const deletedTask = yield* tasks.deleted.get({ taskId: "t-1" })
// deletedTask.title → "Design the API"
```

### Restore

Restore recomposes all keys, increments the version, and brings the item back to life:

```typescript title="restore.ts"
const restored = yield* tasks.restore({ taskId: "t-1" })
// restored.title → "Design the API"
// restored.status → "done"

// Back in GSI queries
const tasksAfterRestore = yield* tasks.collect(Tasks.query.byUser({ userId: "u-alice" }))
// t-1 is back
```

### Purge

Purge permanently removes everything for a partition key -- the current item, all version snapshots, and any soft-deleted copies:

```typescript title="purge.ts"
yield* users.purge({ userId: "u-bob" })

// Completely gone -- no item, no versions, no snapshots
yield* users.get({ userId: "u-bob" })
  .pipe(Effect.catchTag("ItemNotFound", () => Effect.succeed("completely gone")))
```

Purge is the nuclear option. Use it for GDPR "right to erasure" or test cleanup.

### Layer Setup

```typescript title="main.ts"
import { Effect, Layer } from "effect"
import { DynamoClient } from "effect-dynamodb"

const AppLayer = Layer.mergeAll(
  DynamoClient.layer({
    region: "us-east-1",
    endpoint: "http://localhost:8000",
    credentials: { accessKeyId: "local", secretAccessKey: "local" },
  }),
  MainTable.layer({ name: "crud-demo-table" }),
)

const main = program.pipe(Effect.provide(AppLayer))

Effect.runPromise(main).then(
  () => console.log("Done."),
  (err) => console.error("Failed:", err),
)
```

## Key Takeaways

| Concept | How it's used |
|---------|---------------|
| **Typed client** | `const { Users: users } = yield* DynamoClient.make(MainTable)` resolves dependencies; all ops have `R = never` |
| **Variadic combinator updates** | `users.update(key, Entity.set(...), Entity.expectedVersion(n))` — combinators as rest args |
| **Optimistic locking** | `Entity.expectedVersion(n)` composes with `Entity.set()` as variadic combinators |
| **Tagged error handling** | `Effect.catchTag("ItemNotFound", ...)` for precise recovery — bound methods return `Effect` directly |
| **GSI queries** | `Users.query.byRole(composites)` on entity definition, `users.collect(query)` on bound entity |
| **Transactions** | `transactWrite` for atomic multi-entity creates; `transactGet` for typed tuple reads |
| **Sparse GSI** | Optional composites produce sparse indexes; all-or-none constraint prevents stale keys |
| **Version history** | `versioned: { retain: true }` creates snapshots; `users.getVersion(key, n)` and `Users.versions(key)` for retrieval |
| **Soft delete + restore** | `softDelete: true` archives items; `users.restore(key)` recomposes all keys |
| **Purge** | `users.purge(key)` permanently removes item + versions + soft-deleted copies |
