This guide covers how to define access patterns using indexes, how key generation works, and how to use collections for cross-entity queries.

## Entity Primary Key

Entities define only their **primary key**. GSI access patterns are defined separately via `Collections.make()`.

```typescript region="task-entity" example="guide-indexes.ts"
const Tasks = Entity.make({
  model: Task,
  entityType: "Task",
  primaryKey: {
    pk: { field: "pk", composite: ["taskId"] },
    sk: { field: "sk", composite: [] },
  },
  indexes: {
    byProject: {
      name: "gsi1",
      pk: { field: "gsi1pk", composite: ["projectId"] },
      sk: { field: "gsi1sk", composite: ["priority"] },
    },
    byAssignee: {
      name: "gsi2",
      pk: { field: "gsi2pk", composite: ["assigneeId"] },
      sk: { field: "gsi2sk", composite: ["priority"] },
    },
  },
  timestamps: true,
})
```

### Anatomy of a Primary Key

```typescript
primaryKey: {
  pk: {
    field: "pk",                         // Physical DynamoDB attribute
    composite: ["taskId"],               // Model attributes -> partition key
  },
  sk: {
    field: "sk",
    composite: [],                       // Empty = one item per partition key
  },
}
```

| Property | Type | Required | Description |
|----------|------|----------|-------------|
| `pk.field` | `string` | Yes | Physical DynamoDB attribute name |
| `pk.composite` | `string[]` | Yes | Model attributes composing the partition key |
| `sk.field` | `string` | Yes | Physical DynamoDB attribute name |
| `sk.composite` | `string[]` | Yes | Model attributes composing the sort key |

### Logical Names as API Surface

Collection names become query accessors on the typed client:

```typescript region="query-accessors" example="guide-indexes.ts"
// Logical names become query accessors — no physical index names in code
const alphaTasks = yield* basic.entities.Tasks.byProject({ projectId: "proj-alpha" }).collect()

const aliceTasks = yield* basic.entities.Tasks.byAssignee({ assigneeId: "emp-alice" }).collect()
```

You never reference physical index names (`gsi1`, `gsi2`) in application code.

## Key Generation

effect-dynamodb generates DynamoDB key values automatically from your composite declarations. You declare *which* attributes compose each key — the system handles *how*.

### Format

```
{schema}#{version}#{prefix}#{attr1}#{attr2}
```

Given `DynamoSchema({ name: "myapp", version: 1 })` and `entityType: "Task"`:

| Composite | Generated Key |
|-----------|---------------|
| `["taskId"]` with value `"t-001"` | `$myapp#v1#task#t-001` |
| `["projectId", "status"]` with values `"proj-alpha"`, `"active"` | `$myapp#v1#task#proj-alpha#active` |
| `[]` (empty) | `$myapp#v1#task` |

### Empty Composites

Both `pk.composite` and `sk.composite` can be empty arrays. The entity type prefix guarantees no collisions between entity types.

| pk | sk | Use Case |
|----|----|----------|
| `[]` | `[]` | Singleton (config, counters) |
| `["userId"]` | `[]` | One item per user |
| `[]` | `["userId"]` | All users in one partition |
| `["tenantId"]` | `["userId"]` | Users partitioned by tenant |

### Attribute Serialization

| Type | Key Value |
|------|-----------|
| `string` | As-is |
| `number` | Zero-padded |
| `DateTime.Utc` | ISO 8601 |
| `boolean` | `"true"` / `"false"` |
| Branded string | Underlying value |

### Casing Resolution

Structural key parts (schema name, entity type, collection name) follow casing rules. Resolution order:

1. Index-level `casing` (highest priority)
2. Schema-level `casing` (default: `"lowercase"`)

Attribute values are always preserved as-is.

```typescript
// With casing: "lowercase" and entityType: "Employee"
// PK: $myapp#v1#employee#emp-Alice
//     ^^^^^^^^ ^^ ^^^^^^^^          <- structural (lowercased)
//                          ^^^^^^^^^ <- attribute value (preserved)
```

## Collections

Collections group multiple entity types that share a partition key, enabling cross-entity queries. Two modes are supported: **isolated** (default) and **clustered**.

### Isolated Collections (Default)

Each entity owns its sort key prefix. Collection queries use only the partition key — no sort key condition.

```typescript region="isolated-entities" example="guide-indexes.ts"
const IsolatedEmployees = Entity.make({
  model: Employee,
  entityType: "Employee",
  primaryKey: {
    pk: { field: "pk", composite: ["employeeId"] },
    sk: { field: "sk", composite: [] },
  },
  indexes: {
    departmentStaff: {
      collection: "departmentStaff",
      name: "gsi1",
      pk: { field: "gsi1pk", composite: ["department"] },
      sk: { field: "gsi1sk", composite: ["hireDate"] },
    },
  },
  timestamps: true,
})

class Equipment extends Schema.Class<Equipment>("Equipment")({
  equipmentId: Schema.String,
  department: Schema.String,
  name: Schema.String,
  purchaseDate: Schema.String,
}) {}

const Equipments = Entity.make({
  model: Equipment,
  entityType: "Equipment",
  primaryKey: {
    pk: { field: "pk", composite: ["equipmentId"] },
    sk: { field: "sk", composite: [] },
  },
  indexes: {
    departmentStaff: {
      collection: "departmentStaff",
      name: "gsi1",
      pk: { field: "gsi1pk", composite: ["department"] },
      sk: { field: "gsi1sk", composite: ["purchaseDate"] },
    },
  },
  timestamps: true,
})
```

```typescript region="isolated-collection" example="guide-indexes.ts"
```

**Sort key format (isolated):**

```
$myapp#v1#employee_1#2020-01-15     <- Employee entity owns its SK
$myapp#v1#equipment_1#2023-06-01    <- Equipment entity owns its SK
```

**Query behavior:**

| Query | Mechanism | Returns |
|-------|-----------|---------|
| Everything | PK only (no SK condition) | Employee + Equipment |
| Employees only | PK + `begins_with(SK, "$myapp#v1#employee_1")` | Employee |

**Best for:** High-volume single-entity queries where cross-entity queries are occasional.

### Clustered Collections

The collection name sits at the top of every entity's sort key. All entities share this prefix.

```typescript region="clustered-entities" example="guide-indexes.ts"
const ClusteredEmployees = Entity.make({
  model: Employee,
  entityType: "Employee",
  primaryKey: {
    pk: { field: "pk", composite: ["employeeId"] },
    sk: { field: "sk", composite: [] },
  },
  indexes: {
    tenantMembers: {
      collection: "tenantMembers",
      name: "gsi1",
      pk: { field: "gsi1pk", composite: ["tenantId"] },
      sk: { field: "gsi1sk", composite: ["department", "hireDate"] },
      type: "clustered",
    },
  },
  timestamps: true,
})

const ClusteredTasks = Entity.make({
  model: Task,
  entityType: "Task",
  primaryKey: {
    pk: { field: "pk", composite: ["taskId"] },
    sk: { field: "sk", composite: [] },
  },
  indexes: {
    tenantMembers: {
      collection: "tenantMembers",
      name: "gsi1",
      pk: { field: "gsi1pk", composite: ["tenantId"] },
      sk: { field: "gsi1sk", composite: ["projectId", "taskId"] },
      type: "clustered",
    },
  },
  timestamps: true,
})
```

```typescript region="clustered-collection" example="guide-indexes.ts"
```

**Sort key format (clustered):**

```
$myapp#v1#tenantmembers#employee_1#engineering#2020-01-15
$myapp#v1#tenantmembers#task_1#proj-alpha#t-001
^^^^^^^^^^^^^^^^^^^^^^^^^^
collection prefix (shared)
```

**Query behavior:**

| Query | Mechanism | Returns |
|-------|-----------|---------|
| Everything | PK + `begins_with(SK, "$myapp#v1#tenantmembers")` | Employee + Task |
| Employees only | PK + `begins_with(SK, "$myapp#v1#tenantmembers#employee_1")` | Employee |
| Tasks only | PK + `begins_with(SK, "$myapp#v1#tenantmembers#task_1")` | Task |

**Best for:** Cross-entity queries, relationship-dense data.

### Sub-Collections (Isolated Only)

Sub-collections create a hierarchy within the sort key. Multiple collections share the same index, with members grouped at different depths:

```typescript region="subcollection-entities" example="guide-indexes.ts"
const SubEmployee = Entity.make({
  model: Employee,
  entityType: "Employee",
  primaryKey: {
    pk: { field: "pk", composite: ["employeeId"] },
    sk: { field: "sk", composite: [] },
  },
  indexes: {
    contributions: {
      collection: "contributions",
      name: "gsi2",
      pk: { field: "gsi2pk", composite: ["employeeId"] },
      sk: { field: "gsi2sk", composite: ["department"] },
    },
  },
  timestamps: true,
})

const SubTasks = Entity.make({
  model: Task,
  entityType: "Task",
  primaryKey: {
    pk: { field: "pk", composite: ["taskId"] },
    sk: { field: "sk", composite: [] },
  },
  indexes: {
    assignments: {
      collection: "assignments",
      name: "gsi2",
      pk: { field: "gsi2pk", composite: ["employeeId"] },
      sk: { field: "gsi2sk", composite: ["projectId", "taskId"] },
    },
  },
  timestamps: true,
})

const SubProjectMembers = Entity.make({
  model: ProjectMember,
  entityType: "ProjectMember",
  primaryKey: {
    pk: { field: "pk", composite: ["employeeId", "projectId"] },
    sk: { field: "sk", composite: [] },
  },
  indexes: {
    assignments: {
      collection: "assignments",
      name: "gsi2",
      pk: { field: "gsi2pk", composite: ["employeeId"] },
      sk: { field: "gsi2sk", composite: ["projectId"] },
    },
  },
  timestamps: true,
})
```

```typescript region="subcollection-collections" example="guide-indexes.ts"
```

**Sort key hierarchy:**

```
$contributions
+-- #assignments
|   +-- #projectmembers_1#proj-alpha
|   +-- #tasks_1#proj-alpha#t-001
|   +-- #tasks_1#proj-beta#t-002
+-- #employees_1#engineering
```

**Query at any depth:**

| Collection | SK condition | Returns |
|------------|-------------|---------|
| `contributions` | `begins_with "$contributions"` | Employee + Task + ProjectMember |
| `assignments` | `begins_with "$contributions#assignments"` | Task + ProjectMember |
| Single entity | `begins_with "$contributions#assignments#tasks_1"` | Task only |

### Defining Collections

Collections are defined with `Collections.make()`. Each collection owns the full GSI definition (index name, PK field/composites, SK field) and each member entity specifies only its SK composites:

```typescript
const MyCollection = Collections.make({ name: "collectionName",
  index: "gsi1",                    // Physical GSI name
  pk: { field: "gsi1pk", composite: ["tenantId"] },  // Shared PK
  sk: { field: "gsi1sk" },         // Shared SK field
  type: "clustered",               // "clustered" (default) or "isolated"
  members: {
    Employees: Collections.member(EmployeeEntity, {
      sk: { composite: ["department", "hireDate"] },
    }),
    Tasks: Collections.member(TaskEntity, {
      sk: { composite: ["projectId", "taskId"] },
    }),
  },
})
```

### Validation Rules

- All entities in a collection must share the same PK composite on that index
- All entities sharing an index must agree on the type (no mixing isolated/clustered)
- Sub-collections sharing an index must use the same PK and SK fields
- No duplicate entity types within a single collection

## Worked Example: Multi-Tenant Project Management

### Access Patterns

| Pattern | Collection | PK | SK |
|---------|-----------|----|----|
| Get employee by ID | primary | `employeeId` | -- |
| Get task by ID | primary | `taskId` | -- |
| Employees in tenant | TenantMembers (gsi1) | `tenantId` | `department`, `hireDate` |
| Tasks in tenant | TenantMembers (gsi1) | `tenantId` | `projectId`, `taskId` |
| All tenant items | TenantMembers (gsi1) | `tenantId` | -- |
| Employee by email | ByEmail (gsi2) | `email` | -- |
| Tasks by assignee | ByAssignee (gsi2) | `assigneeId` | `priority` |

### DynamoDB Items

| pk | sk | gsi1pk | gsi1sk | __edd_e__ |
|----|----|--------|--------|------------|
| `$myapp#v1#employee#emp-alice` | `$myapp#v1#employee` | `$myapp#v1#tenantmembers#t-acme` | `$myapp#v1#tenantmembers#employee_1#engineering#2024-01-15` | Employee |
| `$myapp#v1#task#t-001` | `$myapp#v1#task` | `$myapp#v1#tenantmembers#t-acme` | `$myapp#v1#tenantmembers#task_1#proj-alpha#t-001` | Task |
| `$myapp#v1#employee#emp-bob` | `$myapp#v1#employee` | `$myapp#v1#tenantmembers#t-acme` | `$myapp#v1#tenantmembers#employee_1#sales#2023-06-01` | Employee |
