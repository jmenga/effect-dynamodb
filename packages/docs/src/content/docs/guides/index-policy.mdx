<Aside type="caution" title="This 'sparse' is not the SparseMap 'sparse'">
This guide covers `indexPolicy: () => ({ attr: 'sparse' | 'preserve' })` — controlling whether an *update* drops a *GSI* membership when a composite is missing. It is unrelated to the **SparseMap storage primitive** (`storedAs: DynamoModel.SparseMap()`), which controls how a `Schema.Record` field is *physically stored* on disk. The two were spelled the same up to 1.5.0 and that collision was a frequent source of confusion. See [Sparse Maps](/effect-dynamodb/guides/sparse-maps/) if you're looking for the storage primitive.
</Aside>

`Entity.update` and time-series `.append` accept partial payloads. `indexPolicy` lets you state, **per composite attribute**, what an *omission* means for the GSI's membership — and the library treats *explicit clears* (`set({ attr: null | undefined })`) as a separate, unconditional cascade signal.

## Mental model — unified attribute hierarchy

Every GSI is a single ordered hierarchy of composite attributes:

```
[pk_1, ..., pk_n,  sk_1, ..., sk_m]
└─ partition prefix ─┘└─ sort/begins_with extension ─┘
```

PK composites form the partition prefix; SK composites extend it for sort and `begins_with` queries. **Policy is per attribute**, but outcomes roll up to the two stored key fields (`gsiNpk`, `gsiNsk`). Reasoning follows position in the hierarchy:

- Clearing a leaf SK composite is a **demotion** — the item stays at coarser scope.
- Clearing an interior PK composite is a **partition migration** — almost always wrong, hence treated unconditionally as "drop the GSI".

This single-hierarchy framing is the key shift in v2. The pre-1.6 "two halves, mostly independent" framing didn't surface these distinctions and led to silent footguns.

## Three-way payload classification

The runtime distinguishes three states for each composite attribute. Collapsing any two would be a footgun:

| Payload state | What library does | Policy consulted? |
|---|---|---|
| `attr: <value>` | Use the value when composing the key | No |
| `attr: null` *or* `attr: undefined` (explicit clear) | Treat as explicit clear. The two collapse — `null` vs `undefined` was a long-running source of dev confusion. **Cascades unconditionally; policy does not protect an explicit clear.** | No (always cascade) |
| (key omitted from payload) | Defer to policy for this attribute | **Yes** |

This is the core change in v2: **omission is the only ambiguous signal**, so it's the only signal `indexPolicy` resolves. Explicit `null`/`undefined` is now an unambiguous "remove this composite from the key now" instruction.

## Per-attribute outcomes

### Omission (attr not in payload)

| Policy | Composite is in PK | Composite is in SK |
|---|---|---|
| `'sparse'` | REMOVE both keys (item drops from GSI) | REMOVE both keys |
| `'preserve'` (default) | No-op — leave stored GSI key attributes untouched | No-op |

### Explicit clear (`attr: null` or `attr: undefined`)

Policy is bypassed; cascade is unconditional.

| Policy | Composite is in PK | Composite is in SK |
|---|---|---|
| `'sparse'` | REMOVE both keys | REMOVE both keys |
| `'preserve'` | REMOVE both keys (degrades to sparse on PK — see below) | **Truncate `gsiNsk`** at this composite — see [Hierarchical SK Pruning](#hierarchical-sk-pruning) |

### Cascade — `Entity.remove([attr])`

Always REMOVEs both key fields for any GSI containing the named attribute, regardless of policy or position. This is the explicit *full drop* primitive.

### PK/SK asymmetry on `preserve` clear

PK composites form the partition prefix; clearing one and "preserving" everything else would migrate the item to a different partition, silently breaking queries against the prior partition. The library treats `preserve` on a PK composite, when that composite is explicitly cleared, as **degraded to `sparse`**: drop the GSI entirely. The narrow case where you genuinely want to migrate partitions can do so via `Entity.remove([attr])` plus a follow-up update with the new value — but the default refuses to do it silently.

## Declaring a policy

```typescript region="entity" example="guide-index-policy.ts"
const Devices = Entity.make({
  model: Device,
  entityType: "Device",
  primaryKey: {
    pk: { field: "pk", composite: ["channel", "deviceId"] },
    sk: { field: "sk", composite: [] },
  },
  indexes: {
    // byAlert — sparse on alertState.
    // When an ingest event omits alertState (plain telemetry), the policy
    // REMOVEs the GSI keys so the device drops out of the alert view.
    byAlert: {
      name: "gsi1",
      pk: { field: "gsi1pk", composite: ["alertState"] },
      sk: { field: "gsi1sk", composite: ["deviceId"] },
      indexPolicy: () => ({ alertState: "sparse" as const }),
    },
    // byTenant — preserve on tenantId.
    // Ingest writers never touch tenantId. When an ingest-side update fires,
    // the preserve policy leaves the stored gsi2pk/gsi2sk untouched so the
    // device remains queryable via the enrichment writer's tenant assignment.
    byTenant: {
      name: "gsi2",
      pk: { field: "gsi2pk", composite: ["tenantId"] },
      sk: { field: "gsi2sk", composite: ["deviceId"] },
      indexPolicy: () =>
        ({ tenantId: "preserve" as const, deviceId: "preserve" as const }) as const,
    },
  },
  timestamps: true,
})
```

The policy is a function so it can branch on the item's current state (e.g. return `'sparse'` only when `item.status === 'draft'`). If you don't declare `indexPolicy` at all, every composite defaults to `'preserve'`.

## Hierarchical SK pruning

When you explicitly clear a *trailing* SK composite whose `indexPolicy` is `preserve`, the library **truncates `gsiNsk`** at that composite — recomposes the SK from the leading prefix instead of dropping the whole GSI. The PK half is recomposed normally if its composites are all present. The item stays queryable in the GSI, just at a *coarser depth* than before.

This unlocks the **demote-don't-evict** pattern across many real-world hierarchies:

| Domain | SK composite hierarchy | Trailing-clear meaning |
|---|---|---|
| Geographic | `[region, country, city, site]` | Asset leaves a site but stays queryable at city/country/region |
| Org | `[division, department, team, squad]` | Engineer rotates off a squad, stays queryable at team/department/division |
| Workflow | `[stage, subStage, step]` | Approval step retracted; item stays queryable at parent stage |
| Content | `[category, subcategory, tag]` | Leaf tag dropped; item stays in subcategory listings |
| Permission | `[org, project, resource]` | Resource access lost; project-level access preserved |
| Order grouping | `[customerId, orderId]` | After clearing `orderId`, group-by-customer queries via `begins_with(sk, "customer#C#")` still work |

The unifying property: leaf composites are *refinements* of the parent context; the parent context remains queryable after pruning.

### Worked example — geographic asset hierarchy

```typescript
indexes: {
  byLocation: {
    name: 'gsi1',
    pk: { field: 'gsi1pk', composite: ['region'] },
    sk: { field: 'gsi1sk', composite: ['country', 'city', 'site'] },
    indexPolicy: () => ({
      region: 'preserve',
      country: 'preserve',
      city: 'preserve',
      site: 'preserve',
    }),
  },
}

// Initial state: asset is at /us/sf/datacenter-1
// Stored: gsi1sk = "$app#v1#asset#country_us#city_sf#site_datacenter-1"

// Asset leaves the datacenter — clear `site`. Item stays queryable at city level.
yield* db.entities.Assets.update(key).set({ site: null })
// gsi1sk truncated to "$app#v1#asset#country_us#city_sf"
// begins_with(gsi1sk, "$app#v1#asset#country_us#city_sf") still finds this asset.

// Asset leaves the city entirely — clear `city` AND `site`.
yield* db.entities.Assets.update(key).set({ city: null, site: null })
// gsi1sk truncated to "$app#v1#asset#country_us"

// Decommission — drop from the index entirely.
yield* db.entities.Assets.update(key).remove(['country'])
// REMOVE gsi1pk, gsi1sk — full drop, cascade overrides truncation.
```

### Hole detection (EDD-9024)

A "hole" is an SK composite cleared at position `i` with another SK composite at position `j > i` still present. Composed keys can't carry holes meaningfully — `acc#A#child#Y` with `parent` cleared composes to a syntactically invalid prefix that no `begins_with` query would match.

The library throws `CompositeKeyHoleError` (EDD-9024) at write time, naming the GSI, the cleared composite at position `i`, and the still-present trailing composite at position `j`:

```typescript
// Hole pattern — throws at write time:
yield* db.entities.Assets.update(key).set({ city: null, site: 'datacenter-2' })
// → CompositeKeyHoleError [EDD-9024]:
//   composite "city" at sk position 1 was cleared, but composite "site" at
//   sk position 2 is still present.
```

To resolve: either supply a value for the cleared composite, clear all trailing composites too, or use `Entity.remove([attr])` to drop the whole GSI explicitly.

## Sparse drop-out (the original `'sparse'`)

Declaring `indexPolicy` opts the GSI into **always-evaluated** semantics. Omission is treated as "not set" per the policy — so a `sparse` composite that isn't in the update payload triggers the drop-out rule:

```typescript region="sparse-drop" example="guide-index-policy.ts"
  // #region sparse-drop
  // Update WITHOUT alertState in the payload. Because byAlert declares an
  // `indexPolicy` with alertState 'sparse', the GSI is always evaluated on
  // every update — alertState absent from payload is treated as "not set"
  // per the policy → REMOVE gsi1pk/gsi1sk. The item drops out of byAlert.
  yield* db.entities.Devices.update({ channel: "c-1", deviceId: "d-1" }).set({ label: "quiet" })

  const afterDrop = yield* db.entities.Devices.byAlert({ alertState: "active" }).collect()
  // →  afterDrop is empty — item dropped out of the alert GSI
```

Re-adding the attr re-indexes the item transparently:

```typescript region="sparse-rehydrate" example="guide-index-policy.ts"
  // #region sparse-rehydrate
  yield* db.entities.Devices.update({ channel: "c-1", deviceId: "d-1" }).set({
    alertState: "cleared",
  })

  const cleared = yield* db.entities.Devices.byAlert({ alertState: "cleared" }).collect()
  // →  cleared contains d-1 under its new alertState
```

## Preserve — the default

With `preserve`, a writer that doesn't own every composite of a GSI can update other attrs without touching the GSI's keys:

```typescript region="hybrid" example="guide-index-policy.ts"
  // #region hybrid
  // Start with an un-indexed device (no tenantId, no alertState).
  yield* db.entities.Devices.put({ channel: "c-2", deviceId: "d-2" })

  // Enrichment writer assigns tenantId.
  yield* db.entities.Devices.update({ channel: "c-2", deviceId: "d-2" }).set({
    tenantId: "initech",
  })

  // Later, ingest writer sets alertState. tenantId is NOT in this payload,
  // but the preserve policy on byTenant leaves its stored keys alone.
  yield* db.entities.Devices.update({ channel: "c-2", deviceId: "d-2" }).set({
    alertState: "active",
  })

  // Both indexes correct — neither writer clobbered the other's composites.
  const finalAlert = yield* db.entities.Devices.byAlert({ alertState: "active" }).collect()
  const finalTenant = yield* db.entities.Devices.byTenant({ tenantId: "initech" }).collect()
  // →  both queries return d-2
```

The ingest writer's `Entity.set({ alertState: "active" })` touches `byAlert` (recomposes its keys) and **also** touches `byTenant` (because `byTenant.sk` has `deviceId`, which is in the primary key → always in the merged payload). Because `tenantId` is `'preserve'` and missing from the payload, the library leaves `byTenant`'s stored keys alone rather than rewriting them with a junk value.

## Cascade precedence

```typescript region="cascade" example="guide-index-policy.ts"
  // #region cascade
  // Regardless of indexPolicy, removing a GSI composite attribute drops the
  // item out of that GSI. Cascade takes precedence over preserve/sparse.
  yield* db.entities.Devices.update({ channel: "c-2", deviceId: "d-2" }).remove(["tenantId"])

  const tenantAfterRemove = yield* db.entities.Devices.byTenant({ tenantId: "initech" }).collect()
  // →  tenantAfterRemove is empty — cascade overrode the preserve policy
```

`Entity.remove(["tenantId"])` always REMOVEs every GSI whose composites include `tenantId`, regardless of `'preserve'` or `'sparse'`. Cascade is the explicit way to opt out of an index entirely; explicit-clear-with-`preserve` on an SK composite is the *demote to coarser scope* primitive.

## Decision rules — full algorithm

For each evaluated GSI, given merged payload `M` (primary key + update payload):

1. **Cascade override.** If any composite is in `removedSet` (named in `Entity.remove([...])`) → REMOVE both keys. Done.
2. **Per-composite outcome:**
   - If attr is in `clearedSet` (explicit `null` or `undefined` in payload): cascade unconditionally — flag for **full drop** if PK or `sparse`-SK; flag for **SK truncation at position i** if `preserve`-SK.
   - Else if attr is present (with value): use value at this position.
   - Else (omitted): consult policy. `sparse` → flag for full drop. `preserve` → no-op for this attribute.
3. **Hole check.** If any attr at hierarchy position `i` is flagged for drop/truncation while an attr at `j > i` is present → throw EDD-9024.
4. **Roll up.**
   - Full drop → REMOVE both keys.
   - SK truncation at position `i` → SET `gsiNsk` to leading prefix `[pk_1..pk_n, sk_1..sk_(i-1)]`. SET `gsiNpk` if all PK composites are present.
   - Else recompose normally — SET each half whose composites are all present, no-op for halves with a missing-preserve composite.

## When is a GSI evaluated?

- **GSI declares `indexPolicy`** → evaluated on **every** update. Policy-first: the declaration is a statement about membership invariant.
- **GSI has no `indexPolicy`** → evaluated only when at least one of its composites is in the update payload (present, or explicitly cleared), or when `Entity.remove([attr])` names one of its composites. Matches conventional partial-update semantics.

## Interaction with `put()`

`put()` does NOT consult `indexPolicy`. It writes a complete item from scratch — any missing composite means "this item is not in that GSI." Unchanged from prior versions.

## Interaction with time-series `.append()`

`indexPolicy` is applied during `.append()` too, with one extra constraint: returned policy keys must be members of `appendInput`. Composites outside `appendInput` are by contract never changed by an append, so their policy can't fire at append-time — they default to `'preserve'` unconditionally, leaving the half that contains them untouched.

This means a GSI like `byAccountAlert` with `pk.composite: ["accountId", "alertState"]` where `accountId` is enrichment-owned (not in `appendInput`) behaves correctly under ingest-side `.append()` calls: the pk half is left alone (accountId preserve, untouched), while any sk composite (e.g. `timestamp`) in `appendInput` is recomposed on every event.

## Quick reference

| Scenario | Policy | Payload | Result |
|---|---|---|---|
| Full recompose | any | all composites present | SET both halves |
| Default partial update | none (all preserve) | some composites missing | SET halves fully present; leave halves with missing attrs alone |
| Sparse omission | `'sparse'` on attr A | A missing | REMOVE both halves |
| Sparse clear | `'sparse'` on attr A | A null/undefined | REMOVE both halves |
| Preserve clear of PK composite | `'preserve'` on PK A | A null/undefined | REMOVE both halves (degrades to sparse on PK) |
| Preserve clear of trailing SK composite | `'preserve'` on SK X | X null/undefined; trailing composites absent | **Truncate `gsiNsk`** to leading prefix; SET pk normally |
| Hole pattern | `'preserve'` on SK X | X null/undefined; trailing composite present | **THROW EDD-9024** |
| `Entity.remove(["A"])` cascade | any | any | REMOVE both halves (cascade overrides) |

## Migrating from 1.5.0

Two semantic changes to be aware of when upgrading:

1. **Null and undefined collapse, both cascade unconditionally.** If you previously relied on `set({ attr: null })` going through `'preserve'` and quietly leaving GSI keys stale, that's now a cascade drop. Audit any null-write paths; intent is now unambiguous. The pre-1.6 behavior was a footgun; 1.6 closes it.
2. **`'sparse'` policies that aren't really membership-driving should switch to `'preserve'`.** The pre-1.6 footgun was: `'sparse'` fired on every update regardless of whether the caller touched that composite. With v2, `'sparse'` still does that on omission — which is correct *only* when the GSI's membership genuinely is driven by that attribute being set. If you used `'sparse'` to mean "drop on actual absence" but always supply the value when you intend the item to stay, your code is fine; if you used it on a hybrid GSI where the attribute is owned by another writer, switch to `'preserve'`.

Hierarchical SK pruning (preserve-clear on trailing SK composite → truncate) is **new and additive**. Existing entities that don't explicitly clear SK composites see no change.
