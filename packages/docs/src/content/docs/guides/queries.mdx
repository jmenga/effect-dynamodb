This guide covers the pipeable Query API: how to construct queries, apply sort key conditions, filter results, paginate, and collect.

## Query\<A\> — A Composable Data Type

Queries in effect-dynamodb are **data types**, not builders. A `Query<A>` is a pure description of what to fetch — no DynamoDB calls happen until you execute it. This follows Effect TS idioms: queries are testable, inspectable, and composable via `pipe`.

```typescript
import { Query, DynamoClient } from "effect-dynamodb"

const { TaskEntity: tasks } = yield* DynamoClient.make(MainTable)

// Construct a query (no execution yet)
const query = TaskEntity.query.byProject({ projectId: "proj-alpha" })

// Compose with combinators
const composed = query.pipe(
  Query.where({ beginsWith: activePrefix }),
  Query.limit(25),
  Query.reverse,
)

// Execute via bound entity (crosses into Effect)
const results = yield* tasks.collect(composed)
```

## Entity Queries

Query accessors live on the **entity definition** (`TaskEntity`), not the bound entity. The accessor argument type is derived from the index's `pk.composite`. The bound entity (`tasks`) provides `get`, `collect`, and `paginate` for execution.

```typescript region="entity-queries" example="guide-queries.ts"
// Primary key lookup (not a query — returns single item)
const task = yield* tasks.get({ taskId: "t-001" })

// Named index queries (return Query<A>) — on the entity definition
const projectTasks = yield* tasks.collect(TaskEntity.query.byProject({ projectId: "proj-alpha" }))
const assigneeTasks = yield* tasks.collect(
  TaskEntity.query.byAssignee({ assigneeId: "emp-alice" }),
)
```

### Generated Accessor Mapping

| Index Definition | Accessor | Argument Type |
|------------------|----------|---------------|
| `primary: { pk: { composite: ["taskId"] }, ... }` | `tasks.get(...)` | `{ taskId: string }` |
| `byProject: { pk: { composite: ["projectId"] }, ... }` | `TaskEntity.query.byProject(...)` | `{ projectId: string }` |
| `byAssignee: { pk: { composite: ["assigneeId"] }, ... }` | `TaskEntity.query.byAssignee(...)` | `{ assigneeId: string }` |

## Sort Key Conditions with `Query.where`

`Query.where` maps to DynamoDB's `KeyConditionExpression` on the **composed sort key string**. It efficiently narrows results at the storage layer (reduces items read from the index).

The condition operates on the full composed sort key value — not on individual composite field names. Use `KeyComposer.composeSortKeyPrefix` to build prefixes from composite attributes.

**beginsWith** — match tasks whose composed sort key starts with a prefix:

```typescript region="begins-with" example="guide-queries.ts"
// beginsWith — match tasks with "active" status prefix in the sort key
const activePrefix = KeyComposer.composeSortKeyPrefix(
  AppSchema,
  "Task",
  1,
  TaskEntity.indexes.byProject,
  { status: "active" },
)
const activeTasks = yield* tasks.collect(
  TaskEntity.query.byProject({ projectId: "proj-alpha" }),
  Query.where({ beginsWith: activePrefix }),
)
```

**between** — inclusive range on the composed sort key:

```typescript region="between" example="guide-queries.ts"
// between — range on the composed sort key
const startDate = "2025-02-01"
const endDate = "2025-02-28"
const lo = KeyComposer.composeSortKeyPrefix(AppSchema, "Task", 1, TaskEntity.indexes.byProject, {
  status: "active",
  createdAt: startDate,
})
const hi = KeyComposer.composeSortKeyPrefix(AppSchema, "Task", 1, TaskEntity.indexes.byProject, {
  status: "active",
  createdAt: endDate,
})
const activeInRange = yield* tasks.collect(
  TaskEntity.query.byProject({ projectId: "proj-alpha" }),
  Query.where({ between: [lo, hi] }),
)
```

**eq** — exact match on the full sort key value:

```typescript
yield* tasks.collect(
  TaskEntity.query.byProject({ projectId: "proj-alpha" }),
  Query.where({ eq: exactSortKeyValue }),
)
```

### SortKeyCondition Type

```typescript
type SortKeyCondition =
  | { eq: string }              // exact equality
  | { lt: string }              // <
  | { lte: string }             // <=
  | { gt: string }              // >
  | { gte: string }             // >=
  | { between: [string, string] }  // inclusive range
  | { beginsWith: string }     // prefix match
```

All values are **composed sort key strings** — use `KeyComposer.composeSortKeyPrefix` to build them from composite attributes.

## Post-Query Filtering

`FilterExpression` is applied after items are read from the index — it does **not** reduce read capacity, only network transfer. Use it for conditions on non-key attributes.

```typescript region="filter-callback" example="guide-queries.ts"
// Callback — type-safe, supports nested paths, OR/NOT, all operators
const highPriActive = yield* tasks.collect(
  TaskEntity.query.byProject({ projectId: "proj-alpha" }),
  TaskEntity.filter((t, { and, eq }) => and(eq(t.status, "active"), eq(t.priority, "high"))),
)
```

```typescript region="filter-shorthand" example="guide-queries.ts"
// Shorthand — simple AND-equality
const activeShorthand = yield* tasks.collect(
  TaskEntity.query.byProject({ projectId: "proj-alpha" }),
  TaskEntity.filter({ status: "active" }),
)
```

**When to use `where` vs `filter`:**

| Combinator | DynamoDB Mapping | Reduces Read Capacity? | Use For |
|------------|------------------|----------------------|---------|
| `Query.where` | `KeyConditionExpression` | Yes | Composed sort key string (prefix, range, exact) |
| `Entity.filter()` | `FilterExpression` | No | Any attribute (post-read) |

See the [Expressions Guide](/effect-dynamodb/guides/expressions/) for the complete operator reference and DynamoDB mapping tables.

## Collection Queries

Collection queries return all member entity types, grouped by entity name.

```typescript
import { Collection } from "effect-dynamodb"

const TenantMembers = Collection.make("TenantMembers", {
  employees: EmployeeEntity,
  tasks: TaskEntity,
})

// All entities in the collection
const all = yield* TenantMembers.query({ tenantId: "t-acme" }).pipe(
  Query.execute,
)
// { employees: Employee[], tasks: Task[] }
```

### Entity Selectors

Narrow a collection query to a single entity type using typed selectors:

```typescript
// Only employees
const employees = yield* TenantMembers.query({ tenantId: "t-acme" }).pipe(
  TenantMembers.employees,
  Query.execute,
)
// Employee[]

// Only employees hired recently
const recent = yield* TenantMembers.query({ tenantId: "t-acme" }).pipe(
  TenantMembers.employees,
  Query.where({ beginsWith: hireDatePrefix }),
  Query.execute,
)
// Employee[]
```

### Sub-Collection Queries

Query at different depths of the collection hierarchy:

```typescript
// All contributions (Employee + Task + ProjectMember)
yield* Contributions.query({ employeeId: "emp-alice" }).pipe(Query.execute)
// { employees: Employee[], tasks: Task[], projectMembers: ProjectMember[] }

// Only assignments (Task + ProjectMember) — sub-collection
yield* Assignments.query({ employeeId: "emp-alice" }).pipe(Query.execute)
// { tasks: Task[], projectMembers: ProjectMember[] }

// Tasks from assignments, filtered by project
yield* Assignments.query({ employeeId: "emp-alice" }).pipe(
  Assignments.tasks,
  TaskEntity.filter({ projectId: "proj-alpha" }),
  Query.execute,
)
// Task[]
```

## Pagination

### Collect All Items

`collect` fetches all pages and flattens into a single array:

```typescript region="collect-all" example="guide-queries.ts"
// Collect all items across all pages
const allProjectTasks = yield* tasks.collect(
  TaskEntity.query.byProject({ projectId: "proj-alpha" }),
)
```

### Single Page

`Query.execute` as a pipe terminal returns a `Page<A>` containing matching items (up to the `limit`) and an optional cursor for pagination:

```typescript region="single-page" example="guide-queries.ts"
// Single page with limit — returns Page<A> with items and cursor
const page = yield* TaskEntity.query
  .byProject({ projectId: "proj-alpha" })
  .pipe(Query.limit(3), Query.execute)
// page.items: Task[] (up to 3 items)
// page.cursor: string | null (pass to Query.startFrom for next page)
```

### Cursor-Based Pagination

Use `Query.startFrom` to resume from a previous page's cursor:

```typescript region="cursor-pagination" example="guide-queries.ts"
// Cursor-based pagination
const page1 = yield* TaskEntity.query
  .byProject({ projectId: "proj-alpha" })
  .pipe(Query.limit(3), Query.execute)

if (page1.cursor) {
  const page2 = yield* TaskEntity.query
    .byProject({ projectId: "proj-alpha" })
    .pipe(Query.limit(3), Query.startFrom(page1.cursor), Query.execute)
  yield* Console.log(
    `Cursor pagination: page1=${page1.items.length} items, page2=${page2.items.length} items`,
  )
}
```

### Streaming

`paginate` returns a `Stream<A>`, automatically handling DynamoDB pagination:

```typescript region="streaming" example="guide-queries.ts"
// Streaming — automatic pagination via Stream
const stream = tasks.paginate(TaskEntity.query.byProject({ projectId: "proj-alpha" }))

const allFromStream = yield* Stream.runCollect(stream)
yield* Console.log(`Stream collect: ${allFromStream.length} tasks\n`)
```

## Scan

The entity definition's `scan()` method reads the entire table (or index) and returns items matching the entity type. It returns a `Query<Entity.Record>`, so all Query combinators and terminals work with scans.

```typescript region="scan-basic" example="guide-queries.ts"
// Basic scan — all items of this entity type
const allTasks = yield* tasks.collect(TaskEntity.scan())
```

```typescript region="scan-filter" example="guide-queries.ts"
// Scan with filter
const activeScan = yield* tasks.collect(
  TaskEntity.scan(),
  TaskEntity.filter({ status: "active" }),
)
```

```typescript region="scan-limit" example="guide-queries.ts"
// Scan with limit
const firstPage = yield* tasks.collect(TaskEntity.scan(), Query.limit(3))
```

```typescript region="scan-consistent" example="guide-queries.ts"
// Scan with consistent read
const consistent = yield* tasks.collect(TaskEntity.scan(), Query.consistentRead)
```

```typescript region="scan-stream" example="guide-queries.ts"
// Stream-based scan
const scanStream = tasks.paginate(TaskEntity.scan())
yield* Stream.runForEach(scanStream, (t) => Console.log(`  Scanned: ${t.taskId} — "${t.title}"`))
```

**When to use Scan vs Query:**

| | Query | Scan |
|---|-------|------|
| Targets | Specific partition | Entire table |
| Efficiency | Reads only matching partition | Reads every item |
| Cost | Low (proportional to results) | High (proportional to table size) |
| Use cases | Normal application queries | Admin tools, migrations, data exports, analytics |

Scan automatically filters by `__edd_e__` — even in a single-table design, `ProductEntity.scan()` only returns Product items.

## Consistent Reads

By default, DynamoDB reads are eventually consistent. For strong consistency, use `consistentRead`:

```typescript region="consistent-get" example="guide-queries.ts"
// Consistent read on get — use entity definition's get + pipe
const consistentTask = yield* TaskEntity.get({ taskId: "t-001" }).pipe(Entity.consistentRead())
```

```typescript region="consistent-query" example="guide-queries.ts"
// Consistent read on query
const consistentQuery = yield* tasks.collect(
  TaskEntity.query.byProject({ projectId: "proj-alpha" }),
  Query.consistentRead,
)
```

```typescript region="consistent-scan" example="guide-queries.ts"
// Consistent read on scan
const consistentScan = yield* tasks.collect(TaskEntity.scan(), Query.consistentRead)
```

Consistent reads cost 2x the read capacity of eventually-consistent reads. Use them when you need read-after-write consistency (e.g., immediately after a put or update).

## Ordering

By default, results are in ascending sort key order. Use `Query.reverse` for descending:

```typescript region="reverse" example="guide-queries.ts"
// Most recent tasks first (descending sort key order)
const recent = yield* tasks.collect(
  TaskEntity.query.byProject({ projectId: "proj-alpha" }),
  Query.reverse,
  Query.limit(3),
)
```

## Complete Example

```typescript
import { Effect, Layer, Stream } from "effect"
import { Collection, Query, DynamoClient } from "effect-dynamodb"

const program = Effect.gen(function* () {
  const { TaskEntity: tasks } = yield* DynamoClient.make(MainTable)

  // --- Single entity query with sort key condition ---
  const activeTasks = yield* tasks.collect(
    TaskEntity.query.byProject({ projectId: "proj-alpha" }),
    Query.where({ beginsWith: activePrefix }),
    Query.limit(50),
  )

  // --- Single entity query, reversed, with filter ---
  const recentHighPriority = yield* tasks.collect(
    TaskEntity.query.byAssignee({ assigneeId: "emp-alice" }),
    TaskEntity.filter({ priority: "high", title: { contains: "API" } }),
    Query.reverse,
    Query.limit(10),
  )

  // --- Collection query: all tenant members ---
  const allMembers = yield* TenantMembers.query({ tenantId: "t-acme" }).pipe(
    Query.execute,
  )
  // { employees: Employee[], tasks: Task[] }

  // --- Collection query: narrow to employees, stream ---
  const employeeStream = yield* TenantMembers.query({ tenantId: "t-acme" }).pipe(
    TenantMembers.employees,
    EmployeeEntity.filter({ department: "Engineering" }),
    Query.paginate,
  )
  yield* Stream.runForEach(employeeStream, (emp) =>
    Effect.log(`Employee: ${emp.displayName}`)
  )
})

const main = program.pipe(
  Effect.provide(
    Layer.mergeAll(
      DynamoClient.layer({ region: "us-east-1" }),
      MainTable.layer({ name: "Main" }),
    )
  )
)
```
