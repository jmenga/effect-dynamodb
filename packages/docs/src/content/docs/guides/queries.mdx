This guide covers the Query API: how to construct queries, apply sort key conditions, filter results, paginate, and collect.

## BoundQuery — A Fluent Chain

Queries in effect-dynamodb are **fluent chains** with pre-resolved services. A collection query or entity scan returns a `BoundQuery` that you chain combinators onto, then terminate with `.collect()`, `.fetch()`, or `.paginate()`.

```typescript
import { Collections, DynamoClient } from "effect-dynamodb"

const db = yield* DynamoClient.make({
  entities: { TaskEntity },
  collections: { ByProject },
})

// Construct and execute a query in one chain
const { Tasks: results } = yield* db.collections
  .ByProject({ projectId: "proj-alpha" })
  .filter({ status: "active" })
  .limit(25)
  .reverse()
  .collect()
```

## Entity Queries

Query accessors live on `db.collections` (for GSI queries) and `db.entities` (for primary key lookups and scans). Collection queries return grouped results; entity operations return single items.

```typescript region="entity-queries" example="guide-queries.ts"
// Primary key lookup (not a query — returns single item)
const task = yield* tasks.get({ taskId: "t-001" })

// Named collection queries — on the db.collections gateway
const { Tasks: projectTasks } = yield* db.collections
  .ByProject({ projectId: "proj-alpha" })
  .collect()
const { Tasks: assigneeTasks } = yield* db.collections
  .ByAssignee({ assigneeId: "emp-alice" })
  .collect()
```

### Generated Accessor Mapping

| Definition | Accessor | Argument Type |
|-----------|----------|---------------|
| `primaryKey: { pk: { composite: ["taskId"] }, ... }` | `tasks.get(...)` | `{ taskId: string }` |
| `Collections.make("ByProject", { pk: { composite: ["projectId"] }, ... })` | `db.collections.ByProject(...)` | `{ projectId: string }` |
| `Collections.make("ByAssignee", { pk: { composite: ["assigneeId"] }, ... })` | `db.collections.ByAssignee(...)` | `{ assigneeId: string }` |

## Post-Query Filtering

`.filter()` applies a `FilterExpression` after items are read from the index -- it does **not** reduce read capacity, only network transfer. Use it for conditions on non-key attributes.

```typescript region="filter-callback" example="guide-queries.ts"
// Shorthand — AND-equality on multiple fields
const { Tasks: highPriActive } = yield* db.collections
  .ByProject({ projectId: "proj-alpha" })
  .filter({ status: "active", priority: "high" })
  .collect()
```

```typescript region="filter-shorthand" example="guide-queries.ts"
// Shorthand — simple AND-equality
const { Tasks: activeShorthand } = yield* db.collections
  .ByProject({ projectId: "proj-alpha" })
  .filter({ status: "active" })
  .collect()
```

**When to use SK composites vs `filter`:**

| Approach | DynamoDB Mapping | Reduces Read Capacity? | Use For |
|----------|------------------|----------------------|---------|
| SK composites in accessor | `KeyConditionExpression` | Yes | Narrowing by sort key prefix |
| `.filter()` | `FilterExpression` | No | Any attribute (post-read) |

See the [Expressions Guide](/effect-dynamodb/guides/expressions/) for the complete operator reference and DynamoDB mapping tables.

## Collection Queries

Collection queries return all member entity types, grouped by member name.

```typescript
const db = yield* DynamoClient.make({
  entities: { EmployeeEntity, TaskEntity },
  collections: { TenantMembers },
})

// All entities in the collection
const { employees, tasks } = yield* db.collections
  .TenantMembers({ tenantId: "t-acme" })
  .collect()
// employees: Employee[], tasks: Task[]
```

### Sub-Collection Queries

Query at different depths of the collection hierarchy:

```typescript
// All contributions (Employee + Task + ProjectMember)
const all = yield* db.collections
  .Contributions({ employeeId: "emp-alice" })
  .collect()
// { employees: Employee[], tasks: Task[], projectMembers: ProjectMember[] }

// Only assignments (Task + ProjectMember) — sub-collection
const assignments = yield* db.collections
  .Assignments({ employeeId: "emp-alice" })
  .collect()
// { tasks: Task[], projectMembers: ProjectMember[] }
```

## Pagination

### Collect All Items

`.collect()` fetches all pages and flattens into a single array (or grouped result for collections):

```typescript region="collect-all" example="guide-queries.ts"
// Collect all items across all pages
const { Tasks: allProjectTasks } = yield* db.collections
  .ByProject({ projectId: "proj-alpha" })
  .collect()
```

### Single Page

`.fetch()` returns a page containing matching items (up to the `limit`) and an optional cursor for pagination:

```typescript region="single-page" example="guide-queries.ts"
// Single page with limit — returns page with items and cursor
const page = yield* db.collections.ByProject({ projectId: "proj-alpha" }).limit(3).fetch()
// page.items: { Tasks: Task[] } (up to 3 items)
// page.cursor: string | null (pass to startFrom for next page)
```

### Cursor-Based Pagination

Use `.startFrom()` to resume from a previous page's cursor:

```typescript region="cursor-pagination" example="guide-queries.ts"
// Cursor-based pagination
const page1 = yield* db.collections.ByProject({ projectId: "proj-alpha" }).limit(3).fetch()

if (page1.cursor) {
  const page2 = yield* db.collections
    .ByProject({ projectId: "proj-alpha" })
    .limit(3)
    .startFrom(page1.cursor)
    .fetch()
}
```

### Streaming

`.paginate()` returns a `Stream<A>`, automatically handling DynamoDB pagination:

```typescript region="streaming" example="guide-queries.ts"
// Streaming — automatic pagination via Stream
const stream = tasks.scan().paginate()
const allFromStream = yield* Stream.runCollect(stream)
```

## Scan

Entity scans read the entire table and return items matching the entity type. `db.entities.Entity.scan()` returns a `BoundQuery`, so all combinators and terminals work with scans.

```typescript region="scan-basic" example="guide-queries.ts"
// Basic scan — all items of this entity type
const allTasks = yield* tasks.scan().collect()
```

```typescript region="scan-filter" example="guide-queries.ts"
// Scan with filter
const activeScan = yield* tasks.scan().filter({ status: "active" }).collect()
```

```typescript region="scan-limit" example="guide-queries.ts"
// Scan with limit
const firstPage = yield* tasks.scan().limit(3).collect()
```

```typescript region="scan-consistent" example="guide-queries.ts"
// Scan with consistent read
const consistent = yield* tasks.scan().consistentRead().collect()
```

```typescript region="scan-stream" example="guide-queries.ts"
// Stream-based scan
const scanStream = tasks.scan().paginate()
yield* Stream.runForEach(scanStream, (t) => Console.log(`  Scanned: ${t.taskId} — "${t.title}"`))
```

**When to use Scan vs Query:**

| | Query | Scan |
|---|-------|------|
| Targets | Specific partition | Entire table |
| Efficiency | Reads only matching partition | Reads every item |
| Cost | Low (proportional to results) | High (proportional to table size) |
| Use cases | Normal application queries | Admin tools, migrations, data exports, analytics |

Scan automatically filters by `__edd_e__` -- even in a single-table design, `Products.scan()` only returns Product items.

## Consistent Reads

By default, DynamoDB reads are eventually consistent. For strong consistency, use `consistentRead`:

```typescript region="consistent-get" example="guide-queries.ts"
// Consistent read on get — use entity definition's get + pipe
const consistentTask = yield* TaskEntity.get({ taskId: "t-001" }).pipe(Entity.consistentRead())
```

```typescript region="consistent-query" example="guide-queries.ts"
// Consistent read on scan (applies to any BoundQuery)
const consistentQuery = yield* tasks.scan().consistentRead().collect()
```

```typescript region="consistent-scan" example="guide-queries.ts"
// Consistent read on scan
const consistentScan = yield* tasks.scan().consistentRead().collect()
```

Consistent reads cost 2x the read capacity of eventually-consistent reads. Use them when you need read-after-write consistency (e.g., immediately after a put or update).

## Ordering

By default, results are in ascending sort key order. Use `.reverse()` for descending:

```typescript region="reverse" example="guide-queries.ts"
// Most recent tasks first (descending sort key order)
const { Tasks: recent } = yield* db.collections
  .ByProject({ projectId: "proj-alpha" })
  .reverse()
  .limit(3)
  .collect()
```

## Complete Example

```typescript
import { Effect, Layer, Stream } from "effect"
import { Collections, DynamoClient } from "effect-dynamodb"

const program = Effect.gen(function* () {
  const db = yield* DynamoClient.make({
    entities: { TaskEntity },
    collections: { ByProject, ByAssignee, TenantMembers },
  })

  // --- Single entity scan with filter ---
  const activeTasks = yield* db.entities.TaskEntity.scan()
    .filter({ status: "active" })
    .limit(50)
    .collect()

  // --- Collection query, reversed, with filter ---
  const { Tasks: recentHighPriority } = yield* db.collections
    .ByAssignee({ assigneeId: "emp-alice" })
    .filter({ priority: "high" })
    .reverse()
    .limit(10)
    .collect()

  // --- Collection query: all tenant members ---
  const { employees, tasks } = yield* db.collections
    .TenantMembers({ tenantId: "t-acme" })
    .collect()
  // employees: Employee[], tasks: Task[]

  // --- Scan with streaming ---
  const scanStream = db.entities.TaskEntity.scan()
    .filter({ status: "active" })
    .paginate()
  yield* Stream.runForEach(scanStream, (t) =>
    Effect.log(`Task: ${t.title}`)
  )
})

const main = program.pipe(
  Effect.provide(
    Layer.mergeAll(
      DynamoClient.layer({ region: "us-east-1" }),
      MainTable.layer({ name: "Main" }),
    )
  )
)
```
